# Security Policy

## Supported Versions

| Version | Supported |
|---------|-----------|
| Latest  | Yes       |
| Older   | No        |

Only the latest released version receives security updates.

## Reporting a Vulnerability

If you discover a security vulnerability in larch, please report it responsibly:

1. **Email**: Send details to <zhupanov@yahoo.com>
2. **Do not** open a public GitHub issue for security vulnerabilities
3. Include steps to reproduce the issue and any relevant context

You should receive an acknowledgment within 72 hours. We will work with you to understand the issue and coordinate a fix before any public disclosure.

## Trust Model

Larch is a Claude Code plugin that runs within Claude Code's permission boundary. It does not bypass Claude Code's built-in permission system at the operator level â€” all tool calls (file edits, shell commands, etc.) at the user-facing assistant level go through the standard permission flow. Consumers running in strict-permissions mode must grant both the bare and fully-qualified `Skill(...)` form for each larch skill they invoke â€” see the "Strict-permissions consumers â€” Skill permission entries" subsection in [`docs/configuration-and-permissions.md`](docs/configuration-and-permissions.md#strict-permissions-consumers--skill-permission-entries) for the required allowlist convention, the shadowing caveat (bare names may resolve to project-local skills before reaching the plugin), and the reason `Skill(larch:*)` wildcards do not currently work.

**External tool delegation**: When Codex or Cursor are available, larch delegates tasks to them. For **review, sketches, voting, and dialectic** tasks, these tools run with the same filesystem access as the user; their non-modification of the working tree is a behavioral request, not a mechanical sandbox. `/design` regular mode runs up to 8 external sketch processes (4 Cursor + 4 Codex); quick mode runs up to 2 (1 Cursor + 1 Codex). The trust model is unchanged â€” same scripts, same prompt constraints. `/implement` Step 2 implementation is performed by the main Claude agent using Edit/Write tools; Codex and Cursor participate only as reviewers (Step 5 quick-mode fallback chain, normal-mode `/review` panel), design sketch authors (`/design`), and voters â€” not as implementors. For `/research` â€” which launches external agents directly against `$PWD` â€” the trust framing (mechanical PreToolUse hook on `/research`'s `Edit|Write|NotebookEdit` surface, prompt-only posture for Bash and external agents, residual risks, and scope of the mechanical guarantee) is documented in the [External reviewer write surface in /research](#external-reviewer-write-surface-in-research) subsection below.

### External reviewer write surface in /research

`/research` delegates review tasks to Cursor and Codex when those tools are available; it launches external reviewers directly against the working tree.

- **External Cursor / Codex filesystem access**: Cursor is launched as `cursor agent ... --workspace "$PWD"` and Codex as `codex exec --full-auto -C "$PWD"`; both processes inherit the user's filesystem privileges and can write the working tree. Non-modification is requested in the reviewer prompt only â€” this is a **behavioral constraint** (prompt-enforced), not a sandbox. The hook described below governs only `/research`'s own Claude tool calls; it does not extend to subprocess-spawned external reviewers.

- **`/research` orchestrator: mechanical limits + prompt-enforced caveats**: the `/research` skill's `Edit | Write | NotebookEdit` tool surface is mechanically guarded by the skill-scoped PreToolUse hook `${CLAUDE_PLUGIN_ROOT}/scripts/deny-edit-write.sh`, which matches `Edit|Write|NotebookEdit` and permits the call only when the target path resolves under canonical `/tmp`, denying every other outcome. The `allowed-tools` frontmatter lists `Edit`, `Write`, `NotebookEdit`, and `Skill` â€” it declares the orchestrator's surface but does NOT confine writes by itself; the hook is the sole mechanical enforcer of the `/tmp`-only policy. **The hook's matcher does NOT include `Bash`** â€” Claude's own Bash calls (heredoc writes, `>>` redirects, subprocesses, `git`) are prompt-only constrained. The matcher likewise does not cover `Skill`; `Skill` is deliberately unscoped so `/research` can invoke `/issue` â€” this also admits other skill invocations, so operators relying on the read-only contract for the whole call graph should treat `Skill`'s breadth as a known residual risk and, where possible, constrain plugin/skill visibility at the Claude Code level.

- **Residual risk recap**: the mechanical hook depends on the running Claude Code version honoring `permissionDecision: "deny"` â€” a non-honoring host has no fallback, and the agent can write anywhere the `Edit`/`Write`/`NotebookEdit` tools reach. **Scope of the mechanical guarantee**: the `/research` hook governs only `/research`'s own tool calls â€” child skills invoked via `Skill` run under their own `allowed-tools` and hooks, not under this one. `/issue`'s writes go to `$ISSUE_TMPDIR` under `/tmp` by its own convention (prompt-enforced, not mechanically inherited from the parent). `/tmp` itself is shared scratch, not session-scoped; one skill's tmpdir is readable by other skills invoked in the same session â€” `/tmp` placement is **not a confidentiality boundary**.

- **`/research --scale=quick` lower-assurance mode (#418, #520)**: `--scale=quick` runs **K=3 homogeneous Claude Agent-tool lanes with vote-merge synthesis** (issue #520 evolved this from 1 inline lane) and **skips the Step 2 validation phase entirely** â€” by design, for fastest lookups. This means none of the validation reviewers (including the Code Reviewer subagent that walks the security focus area) run on quick-mode research output. The synthesis carries an explicit "K-lane voting confidence â€” no validation pass; correlated-error risk: all K lanes are Claude" disclaimer on the vote path (`LANES_SUCCEEDED >= 2`), or the existing "single-lane confidence" disclaimer on the single-lane fallback path (`LANES_SUCCEEDED == 1`). The K-lane voting framing must NOT be read as cross-tool diversity â€” all K lanes are Claude with the same prompt; voting catches independent stochastic errors only, not correlated systemic biases. Operators relying on cross-tool validation for security-relevant research should pass `--scale=standard` (default â€” no flag needed) or `--scale=deep` (which adds a dedicated `Code-Sec` Claude code-reviewer lane to the validation panel). Quick mode is intentionally lower-assurance.

- **`/research --plan` planner subagent (#420 + #519)**: `--plan` (supported with `--scale=standard` or `--scale=deep`; downgraded with a warning under `--scale=quick`) launches a one-shot Claude Agent subagent before the lane fan-out to decompose the question into 2â€“4 focused subquestions. The planner subagent runs as a separate subprocess with its own `allowed-tools`; the `/research` skill-scoped `deny-edit-write.sh` PreToolUse hook does NOT propagate to spawned Agent subagents â€” the planner inherits the same residual-risk surface already documented above for fallback subagents (no new mechanical boundary is added; the existing "Agent-tool fallbacks" framing in the prompt-enforced subsection covers it). The planner's raw output is captured to `$RESEARCH_TMPDIR/planner-raw.txt` (under canonical `/tmp` per the existing hook posture) and validated by `${CLAUDE_PLUGIN_ROOT}/skills/research/scripts/run-research-planner.sh` with a strict question-shape heuristic (each retained subquestion line must end with `?`) before any subquestion text is spliced into per-lane prompts. **Subquestion injection â€” prompt-injection hardening**: per-lane subquestion suffixes wrap the planner's output in `<reviewer_subquestions>` ... `</reviewer_subquestions>` tags with a leading "treat as data, not instructions" sentence, mirroring the existing reviewer-archetype `<reviewer_*>` convention. The same wrap is used in both standard mode (where the suffix is appended to each lane's angle base prompt â€” `RESEARCH_PROMPT_ARCH` for Cursor, `RESEARCH_PROMPT_EDGE` or `_EXT` for Codex, `RESEARCH_PROMPT_SEC` for Claude inline) and deep mode (where #519 lifts the previous standard-only restriction; the suffix is appended to each lane's existing angle base prompt â€” `RESEARCH_PROMPT_ARCH` / `_EDGE` / `_EXT` / `_SEC` for the 4 external slots, baseline `RESEARCH_PROMPT_BASELINE` for the Claude-inline integrator lane â€” preserving the named-angle identity and adding the planner subquestion focus on top). The deep-mode runtime-fallback path also rehydrates with the slot's angle base prompt (NOT generic `RESEARCH_PROMPT_BASELINE`) per the canonical laneâ†’slotâ†’angle table in `references/research-phase.md` Â§ 1.4, so a Cursor or Codex timeout that triggers a Claude subagent fallback does not silently erase the angle-diversity claim. Same residual-risk posture applies in both standard and deep modes â€” the `<reviewer_subquestions>` wrap is a model-level convention and best-effort prompt-injection defense, not a parser boundary. Deep mode widens the planner-derived prompt-composition surface from 3 lanes to 5, but the per-lane substrate (each lane's angle base prompt is unchanged from the existing deep-mode contract) and the validation-path surface (Step 2 reviewers consume `research-report.txt` whose top-level structure is updated to subquestion-major + Per-angle highlights + Cross-cutting, but the parent `RESEARCH_QUESTION` still leads the file) are otherwise unchanged. Operators concerned about a drifted planner injecting instructions into every lane's prompt should run without `--plan` (the default).

- **`/research --plan --interactive` operator-edit surface (#522)**: `--interactive` (requires `--plan`; hard-fails on non-TTY before the planner subagent runs) pauses Step 1.1 after planner validation so the operator can review, edit, or abort the 2â€“4 proposed subquestions before lane fan-out. Adds two new surfaces relative to plain `--plan`. (1) **`$EDITOR` subprocess**: the orchestrator launches the operator-controlled `$EDITOR` value to edit `$RESEARCH_TMPDIR/subquestions-edit.txt`. The `$EDITOR` value is intentionally unquoted at invocation to support multi-word values like `code --wait` or `emacsclient -t`, so it goes through the operator's shell word-splitting. **Trust model**: same as the operator's interactive shell â€” `$EDITOR` is operator-controlled, not skill-controlled, and the spawned editor runs with the operator's privileges. A pathological `$EDITOR` value (e.g., one that writes outside `$RESEARCH_TMPDIR`) is out of scope for this skill â€” the variable already carries the operator's full shell trust. The subprocess is bypass mechanically by the `/research` skill-scoped `deny-edit-write.sh` PreToolUse hook (the hook matches `Edit|Write|NotebookEdit`, NOT `Bash`); same residual-risk posture as the existing `--keep-sidecar` `cp` (also Bash-mediated). (2) **Operator-edited subquestion text** feeding the existing per-lane prompt-composition path. Operator edits are re-validated through `run-research-planner.sh` (same `?`-suffix and 2â€“4 count rules as planner output, plus the new `delimiter_collision` `REASON` token rejecting any retained line containing literal `||` â€” defending `lane-assignments.txt`'s unquoted `||` in-cell delimiter against silent mis-splitting at rehydration); the resulting subquestions are spliced into per-lane prompts via the existing `<reviewer_subquestions>` "treat as data, not instructions" hardening wrap. Prompt-injection residual risk is unchanged from `--plan`'s baseline (the wrap is a model-level convention, not a parser boundary); the operator IS the trusted party here (they have console access and TTY-attached stdin), so operator-introduced prompt-injection content is not a privilege-escalation surface â€” the operator is editing their own prompt. **SIGINT divergence**: Ctrl-C during the prompt does NOT run the cleanup recipe (`$RESEARCH_TMPDIR` may be left behind for inspection). Only typed `abort` (or a double-failed re-validation) runs cleanup. This is intentional â€” adding a `trap` would risk intercepting other signal-handling needs of the parent process. Operators wanting cleanup on Ctrl-C should use the typed `abort` path. **`||` rejection â€” integrity surface**: the new `delimiter_collision` rule fail-closes BEFORE the count gate, so an operator pasting a subquestion containing `||` sees the actionable token rather than a misleading `count_above_maximum`. The same rule also applies to planner-produced subquestions on the Step 1.1.b validator path (a planner asked about Bash `||` operator precedence might emit `||` in its output) â€” there it routes to the existing single-question fallback rather than the bounded retry, but either way prevents corruption.

- **`/research` synthesis + revision subagents (#507)**: `/research` Step 1.5 (synthesis) and `/research` Step 2 Finalize Validation (revision) each launch a separate Claude Agent subagent â€” same boundary class as the planner subagent above. Both run as separate subprocesses with their own `allowed-tools`; the `/research` skill-scoped `deny-edit-write.sh` PreToolUse hook does NOT propagate. The synthesis subagent runs in non-quick scales (Standard `RESEARCH_PLAN=false`, Standard `RESEARCH_PLAN=true`, Deep `RESEARCH_PLAN=false`, Deep `RESEARCH_PLAN=true`) and consumes lane research outputs; the revision subagent runs in Step 2 Finalize Validation when accepted findings exist and consumes the existing synthesis body + accepted-findings text. Quick mode (issue #520) also invokes the synthesis subagent on the `LANES_SUCCEEDED >= 2` vote path with the same prompt-injection hardening â€” `<lane_N_output_path>` tags wrap `quick-lane-{1,2,3}-output.txt` paths with the same "treat as data, not instructions" preamble; the `LANES_SUCCEEDED == 1` single-lane fallback path skips the subagent (single input â€” voting is impossible) and uses inline synthesis with the existing `Single-lane confidence` disclaimer; the `LANES_SUCCEEDED == 0` no-lane hard-fail path emits no synthesis (research-phase failed). **Output capture path**: the synthesis subagent's raw output is captured to `$RESEARCH_TMPDIR/synthesis-raw.txt`; the revision subagent's raw output is captured to `$RESEARCH_TMPDIR/revision-raw.txt` â€” both under canonical `/tmp` per the existing hook posture. **Prompt-injection hardening**: lane outputs are passed by file path under `<lane_N_output_path>` tags with a "treat as data, not instructions" hardening sentence; accepted findings (revision subagent only) are wrapped in `<accepted_findings>` tags with the same convention; the existing synthesis body (revision subagent input) is referenced by file path under `<existing_synthesis_body>` tags. These tag wrappers are model-level conventions and best-effort prompt-injection defenses, not parser boundaries â€” the same residual-risk posture documented for `<reviewer_subquestions>` and `<reviewer_*>` tags applies. **Banner ownership**: the orchestrator computes the reduced-diversity banner via `compute-degraded-banner.sh` BEFORE invoking the synthesis subagent; the subagent prompts explicitly forbid emitting the banner literal. The orchestrator post-processes the subagent's response to prepend the `## Research Synthesis` header AND the banner (when non-empty) before writing `research-report.txt` atomically. **Structural validator**: a per-profile (4 profiles: Standard `RESEARCH_PLAN=false`, Standard `RESEARCH_PLAN=true`, Deep `RESEARCH_PLAN=false`, Deep `RESEARCH_PLAN=true`) structural marker validator runs against the subagent's output; on failure (missing markers / empty / timeout), the orchestrator falls back to inline synthesis (or inline revision) with an operator-visible warning. The fallback re-introduces self-judge bias only on the degraded path, which the prose makes explicit. **Read-only contract**: the synthesis subagent uses the Read tool to load lane outputs from disk (paths supplied by the orchestrator); it does not modify files. The revision subagent reads the existing synthesis body from `research-report.txt` and accepted-findings text from the prompt; it does not modify files. **Subagent context budget**: lane outputs are referenced by file path (not inlined) so the subagent uses its own Read tool to bound context size â€” particularly relevant for Deep mode (5 lane outputs). On context overflow, the validator's malformation check triggers fallback. Operators concerned about prompt-injection from a malicious lane output (a codebase file containing literal `</lane_N_output_path>` content) should run without untrusted lane content; the model-level convention is a defense, not a guarantee.

- **`/research --keep-sidecar` opt-in workspace write (#510)**: `--keep-sidecar` (boolean form) and `--keep-sidecar=<PATH>` (value form) preserve a `/issue`-batch markdown sidecar of findings past Step 4 cleanup. The default destination is `./research-findings-batch.md` (the operator's current working directory); an explicit path is honored when supplied via the `=<PATH>` form. The implementation uses a Bash `cp` and is therefore **prompt-only constrained** (the `deny-edit-write.sh` hook matches `Edit|Write|NotebookEdit` and does NOT match `Bash`; see the residual-risk recap above). The flag is opt-in by the operator â€” passing it constitutes explicit consent to the workspace write. Path validation runs BEFORE the `cp`: the destination MUST NOT resolve under `$RESEARCH_TMPDIR` (canonicalized via `realpath` when available, falling back to a string-prefix check after `cd "$(dirname â€¦)" && pwd`); the destination's parent must exist and be writable. Destinations failing validation produce a warning and skip the `cp` rather than aborting the run. **Operator content-review responsibility**: the sidecar carries the rendered findings prose verbatim from synthesis. With `/research --scale=deep`, the synthesis may include findings from the dedicated `Codex-Sec` security lane â€” operators should review the sidecar (and apply additional redaction where needed) BEFORE filing GitHub issues from it. To make this discipline tractable, SKILL.md Step 4's post-cleanup advertisement deliberately recommends `/issue --input-file <path> --label research --dry-run` rather than `--go`, so the operator manually escalates after review. **Concurrent-run overwrite**: the default destination is shared across runs; two `/research --keep-sidecar` invocations racing in the same working directory will clobber each other unconditionally. Operators automating multiple concurrent runs should pass distinct `--keep-sidecar=<PATH>` values. **Helper contract**: `${CLAUDE_PLUGIN_ROOT}/skills/research/scripts/render-findings-batch.sh` (sibling contract `render-findings-batch.md`) consumes `$RESEARCH_TMPDIR/research-report-final.md` and emits `$RESEARCH_TMPDIR/research-findings-batch.md`; the round-trip integration test through `skills/issue/scripts/parse-input.sh` is wired into `make lint` via `test-render-findings-batch`. Compose-time content (Risk / Difficulty / Feasibility / Files-touched) is global per-finding, NOT per-finding precision â€” the audit-context italic line at each item's tail signals the prose-derivation origin to mitigate misreading.

**Consumers running in strict-permissions environments** that depend on `/research`'s read-only-repo contract should validate hook behavior on their target Claude Code version and, where available, pin `Skill(...)` / `Edit` / `Write` / `NotebookEdit` permissions narrowly. **`Bash(...)` is the residual mechanical bypass on `/research`** â€” the hook does not match `Bash`, so a `Bash`-mediated repo write cannot be prevented by the skill-scoped hook alone. Operators relying on the read-only contract should also pin `Bash(...)` permissions narrowly at the Claude Code level (per `docs/configuration-and-permissions.md`); if a project cannot meaningfully constrain `Bash`, the read-only posture for `/research` is best-effort behavioral, not mechanical. See `skills/research/SKILL.md` ("Read-only-repo contract") for the skill-side framing and [`docs/review-agents.md`](docs/review-agents.md#external-reviewer-trust-boundary-skills-using-cursor--codex-against-pwd) for the skill-author-facing summary.

**Slack tokens**: If configured, `LARCH_SLACK_BOT_TOKEN` is used to post PR announcements. The token should have minimal OAuth scopes (`chat:write`, `reactions:write`). Never commit tokens to version control.

**Layered secret scanning**: larch runs pattern- and verification-based secret scanners at three layers. The layers enforce different guarantees, and conflating them leads to false-confidence gaps â€” especially around `make lint-only` and `/relevant-checks`, which are pre-commit-driven and therefore depend on the pre-commit hook's actual scan scope.

- **Layer 1 â€” commit-time working-tree scan (opt-in via `pre-commit install`)**: `.pre-commit-config.yaml` registers `gitleaks` with an `entry` override to `gitleaks detect --source . --no-git --redact --no-banner`. `--no-git` is load-bearing: without it, `gitleaks detect` scans only the git log and silently passes secrets that are staged but not yet committed. With `--no-git`, the hook scans the working-tree filesystem â€” which includes both tracked and staged content at commit time â€” and blocks the commit on any finding. Hook `pass_filenames: false` means `pre-commit run --files <paths>` still triggers a full-tree scan, so scoped paths like `/relevant-checks` cannot silently miss a secret that lives outside the changed file set. This layer is bypassable via `git commit --no-verify`; Layer 2 is the enforced backstop. Git-history scanning is explicitly NOT Layer 1's job â€” that is Layer 2.
- **Layer 2 â€” PR gate, git-history scan (CI)**: `.github/workflows/ci.yaml` defines a dedicated `gitleaks` job that checks out with `fetch-depth: 0`, installs the same pinned `v8.18.4` engine used by pre-commit (downloaded directly from the gitleaks release), and runs `gitleaks detect --source .` (without `--no-git`) so it walks the full git log. This is complementary to, not redundant with, the `lint` job (which runs the pre-commit hook in working-tree mode): together they cover the whole codebase in both dimensions â€” current working-tree state AND complete history â€” with a single pinned engine version.
- **Layer 3 â€” PR gate, live verification (CI)**: the `trufflehog` job pins `trufflesecurity/trufflehog` to its commit SHA for `v3.82.13` (tags are mutable â€” a force-pushed upstream tag could silently swap a security scanner's binary; SHAs are immutable) with `version: 3.82.13` pinning the Docker image and `--only-verified`, meaning findings fire only for credentials that actually authenticate against a live provider API. This is non-redundant with gitleaks: gitleaks catches regex-matched patterns including synthetic tokens; trufflehog catches ONLY exploitable live secrets. A finding in any layer blocks the PR.

`.gitleaks.toml` maintains a narrow path-based allowlist for four purposes only: the config's self-allowlist (`^\.gitleaks\.toml$`), redaction-scanner source (`scripts/redact-secrets.sh`), redaction-scanner test fixtures (`scripts/test-redact-secrets.sh`), and the tracking-issue-write lifecycle shell helper plus its contract doc and regression harness (`scripts/tracking-issue-write.sh`, `scripts/tracking-issue-write.md`, `scripts/test-tracking-issue-write.md`). These paths legitimately carry token-shaped strings throughout â€” regex literals, token-family tables, and synthetic test inputs â€” so per-line allowlisting would churn without adding signal. **High-churn documentation is NOT allowlisted** (#375): `README.md`, `CHANGELOG.md`, `SECURITY.md` itself, `skills/issue/SKILL.md`, and `skills/issue/scripts/create-one.sh` are scanned by gitleaks in both Layer 1 (pre-commit, working tree) and Layer 2 (CI, full history). The layer responsibilities remain distinct: gitleaks Layers 1â€“2 catch regex-matched patterns including synthetic token-shaped literals in docs; trufflehog Layer 3 (`--only-verified`) catches ONLY live, authenticable credentials and is non-redundant with gitleaks for that reason, NOT a replacement for it â€” an accidental paste of a revoked token or a covered-family token in an unusual format is caught by gitleaks Layers 1â€“2, not by the verified-only scan. Tokens whose format falls outside gitleaks' covered rule families (see the "Outbound shell-layer redaction" subsection below for the covered families) may slip both Layer 1â€“2 (no matching regex rule) and Layer 3 (nothing to authenticate against a live API) â€” contributors must not rely on scanner layers as a substitute for editorial discipline in docs. Contributors adding new token-shaped examples to docs should use non-detector-matching forms: short prefixes without the high-entropy suffix (e.g., `ghp_` as a prefix mention rather than a full 40-character token) or an explicit placeholder like `<REDACTED-TOKEN>`.

**Reviewer archetype security lane**: The Code Reviewer archetype (`agents/code-reviewer.md`, generated from `skills/shared/reviewer-templates.md`) includes a first-class Â§5 Security focus area covering injection, authN/authZ, secret scanning, crypto, deserialization, SSRF, path traversal, and dependency CVEs. Review findings may be tagged `security` as their primary focus area. Reviewer `{CONTEXT_BLOCK}` material (diffs, plans, commits) is wrapped in namespaced `<reviewer_*>` XML tags with a prepended instruction sentence that the tags are literal input delimiters. This is a model-level convention that reduces prompt-injection attack surface; it is NOT a parser-enforced security boundary. A crafted payload inside the content (e.g., a diff line containing a literal matching closing tag) can theoretically defeat the wrapper, and the primary defense is the instruction sentence combined with the namespaced prefix. See `docs/review-agents.md` for the full residual-risk discussion and possible stronger follow-up mitigations.

**Untrusted GitHub Issue Content (`/issue` Phase 2)**: The `/issue` skill performs LLM-based semantic duplicate detection in 2 phases. Phase 1 reads only issue titles from `gh issue list`; Phase 2 fetches full bodies and comments for shortlisted candidates via `${CLAUDE_PLUGIN_ROOT}/skills/issue/scripts/fetch-issue-details.sh`. All fetched content is untrusted â€” issue authors can write arbitrary text including text that looks like instructions. Phase 2 wraps each fetched issue in a per-issue `<external_issue_<N>>â€¦</external_issue_<N>>` block inside an outer `<external_issues_corpus>â€¦</external_issues_corpus>` envelope. New items being created in the same batch are separately wrapped in `<new_item_<i>>â€¦</new_item_<i>>` blocks. Both wrappers are prefaced with a literal instruction that the tags delimit data, not instructions â€” the same model-level convention used by the reviewer archetype. These tags are a prompt-level convention only; they reduce but do not eliminate prompt-injection risk. A crafted payload (e.g., content containing a literal matching closing tag) can theoretically defeat the wrapper, and the primary defense is the instruction sentence combined with the namespaced tag prefix. Phase 2 also applies candidate-ID whitelist validation â€” the LLM may only mark an existing issue as a duplicate if that issue's number appeared in the Phase 1 snapshot; unknown numbers are rejected and fall back to CREATE. This is a correctness guard, not a security sandbox.

**Outbound shell-layer redaction (`/issue` â†’ `gh issue create`)**: `skills/issue/scripts/create-one.sh` pipes both the issue title and the issue body through `scripts/redact-secrets.sh` before invoking `gh issue create`, and also redacts any `gh` stderr captured on the failure-echo path. This is a deterministic defense-in-depth backstop to the prompt-level sanitization rules in `skills/implement/SKILL.md` and `skills/issue/SKILL.md`. Covered token families: Anthropic/OpenAI `sk-*` and `sk-ant-*` keys, GitHub PATs (`ghp_`, `gho_`, `ghu_`, `ghs_`, `ghr_`, `github_pat_`), AWS long-term access keys (`AKIAâ€¦`), Slack tokens (`xoxb-`, `xoxa-`, `xoxp-`, `xoxr-`, `xoxs-`), generic JWTs, and PEM private-key blocks. Explicit non-coverage: AWS STS temporary credentials (`ASIAâ€¦`), payment-provider live keys, opaque bearer tokens without a distinctive prefix, database connection strings, private hostnames, and PII â€” operators should treat this as partial defense-in-depth, not comprehensive secret detection. `scripts/test-redact-secrets.sh` exercises unit, idempotency, and end-to-end (with a stub `gh`) assertions and is wired into `make lint`. Helper failure is fail-closed: `create-one.sh` exits 3 with `ISSUE_ERROR=redaction:â€¦` rather than publishing un-redacted content.

**`/issue --go` approval semantics (single and batch modes)**: The `--go` flag posts `GO` as the final comment on each newly-created issue, which is the approval sentinel that makes an issue eligible for `/fix-issue` automation (see `skills/fix-issue/SKILL.md`). `--go` works in both single and batch modes â€” in batch mode (`--input-file F --go`) it can approve multiple newly-created issues for `/fix-issue` in a single invocation (one `GO` comment per successful CREATE; duplicates, failed creates, and dry-run items never receive a `GO` comment). `gh issue comment` stderr from a failed GO post is piped through `scripts/redact-secrets.sh` before being surfaced in the per-item warning, matching the posture of `create-one.sh`'s outbound redaction. **Operator guidance**: orchestrators that rely on human GO-gating to throttle `/fix-issue` queue growth should restrict which `/issue` callers may pass `--go` (or avoid `--go` in batch mode) â€” a single batch invocation can approve many issues at once. **Fail-open-dedup amplifier**: `/issue`'s 2-phase semantic dedup fails open (any Phase 1 / Phase 2 helper failure â€” network, rate limit, gh auth â€” falls through to create-all); combined with `--go` in batch mode this means a transient dedup-helper failure can produce a burst of newly-created AND auto-approved issues for `/fix-issue`. GO-post failures themselves are non-fatal and audited via per-item `ISSUE_<i>_GO_POSTED=true|false` stdout lines; a `false` value means the issue exists but is NOT yet eligible for `/fix-issue` until a human adds `GO` manually.

**Dev-only PostToolUse audit log (`scripts/audit-edit-write.sh`)**: `scripts/audit-edit-write.sh` is a contributor-local debugging aid that, when opted in via `.claude/settings.local.json`, records a JSONL trail of every `Edit` / `Write` tool invocation to `.claude/hook-audit.log`. The log is **intentionally unredacted** â€” it captures the raw PostToolUse payload including `tool_input` fields like `file_path`, `content`, and `old_string` / `new_string`, which may contain secrets, credentials, PII, or proprietary code. **`/create-skill` description sanitization**: `/create-skill` accepts a free-form `<description>` that gets embedded into the generated skill's YAML frontmatter and into heredoc-in-shell output. `skills/create-skill/scripts/validate-args.sh` rejects descriptions containing patterns that could escape either layer: XML tags (`<...>`), backticks, command-substitution (`$(...)`), heredoc terminators (`EOF`, `HEREDOC`) or YAML frontmatter fences (`---`) as standalone tokens, and any newlines or ASCII control characters. The renderer always double-quotes the description in YAML and backslash-escapes inner quotes. This is defence-in-depth, not a sandbox â€” reviewers should still read generated `SKILL.md` files before accepting PRs. **Two-concept split (#549)**: the orchestrator separates the validated frontmatter `description:` (one-line, `validate-args.sh`-clean) from the freeform feature spec forwarded to `/im` as a feature brief. `skills/create-skill/scripts/prepare-description.sh` (Step 1.5 / Step 1.6 coordinator) probes `validate-args.sh` on the raw input and routes failures to one of three modes: `verbatim` (raw passes â€” used as both frontmatter and brief), `needs-synthesis` (raw failed on the literal `Description contains newlines or control characters` OR `Description length (` validator error â€” orchestrator distills a one-line `Use whenâ€¦` synthesis, applies a name-echo guard, re-validates), or `abort` (any other validator failure). The synthesized line is always re-validated via `validate-args.sh` before reaching `render-skill-md.sh --description`, so the frontmatter sanitization story above (XML / backticks / `$(` / heredoc / control-char rejection + double-quoting + backslash-escaping) holds verbatim for synthesized lines. **Mixed-input pre-synthesis scan**: `prepare-description.sh` rejects mixed inputs (synthesis-trigger class plus any of XML / backtick / `$(` / standalone heredoc-token) as `MODE=abort` regardless of which validator check fires first â€” this prevents a banned-token-bearing original spec from forwarding as the freeform brief while a clean synthesized one-liner takes the frontmatter slot. The freeform brief carries the raw user input verbatim into `/im`'s feature-description block (NOT into a YAML field, NOT into a shell-injection surface) â€” operators should still avoid pasting sensitive content into the `/create-skill` argument because the `/im` brief is consumed by an LLM and may surface in PR bodies, anchor comments, and Slack messages. **Renderer body path (#568)**: `render-skill-md.sh` accepts an optional `--feature-spec-file <path>` whose content is interpolated into the scaffolded SKILL.md body's opening paragraph. The bytes flow from a tmpfile under canonical `/tmp` (`$RAW_DESC_FILE` from `/create-skill` Step 1.4) into a bash variable via `FEATURE_SPEC="$(cat "$FEATURE_SPEC_FILE")"`, then expand once via `${FEATURE_SPEC}` inside an unquoted heredoc, after which the heredoc output is atomically `mv`-d into the committed scaffold. **Heredoc safety**: bash parameter expansion is single-pass â€” substituted file content is data, not re-parsed shell syntax (no nested `${VAR}` / `` ` `` / `$(â€¦)` evaluation), and heredoc terminator scan happens on the literal source code, not on substituted content. **Pre-synthesis scan still applies**: `prepare-description.sh` F9 rejects raw inputs containing XML tags, backticks, `$(`, or standalone heredoc/frontmatter tokens (`EOF` / `HEREDOC` / `---`) BEFORE the orchestrator forwards the path to `--feature-spec-file`, so the body content carries the same banned-token guarantees as the frontmatter on the synthesis path. The body is intentionally NOT YAML-escaped (it lands in markdown prose, not a YAML scalar) and intentionally NOT routed through `validate-args.sh` (multi-line specs are valid by design). **Committed-output surface**: the rendered `SKILL.md` is committed by the implementing agent in the `/im` PR â€” operators should still read the rendered scaffold before merging, same defence-in-depth posture as the pre-#568 frontmatter pipeline.

**`tracking-issue-write.sh` outbound path**: `scripts/tracking-issue-write.sh` (umbrella #348 Phase 1 foundation helper) mirrors `create-one.sh`'s fail-closed redaction posture for a new shell-layer lifecycle surface (`create-issue`, `append-comment`, `upsert-anchor` subcommands). Three load-bearing invariants document the security contract. **Redaction order** â€” every subcommand composes the full logical body in memory, pipes it through `scripts/redact-secrets.sh`, and only then applies truncation. Reversing the order could slice token-shaped byte sequences and let secrets past the scrubber; the placement is a structural choke point in code (mirrors the `create-one.sh:202-208` "Single structural choke point" comment). **gh-failure redaction** â€” every `gh` invocation captures stdout and stderr separately; on non-success paths the captured stderr is piped through `scripts/redact-secrets.sh` before emission in the `ERROR=` line. This covers the 4xx API response body leak surface (tokens echoed in request-body error responses) that `create-one.sh:247-280` already addresses for `/issue`. **Anchor-skeleton preservation** â€” truncation operates on section interiors only, never on the HTML first-line anchor marker (`<!-- larch:implement-anchor v1 issue=<N> -->`) or any `<!-- section:<id> -->` / `<!-- section-end:<id> -->` marker literal. Phase 3 consumers parse by these markers; corrupting them would break downstream parsers silently. Helper failure is fail-closed: the script exits 3 with `FAILED=true ERROR=redaction:â€¦` rather than publishing un-redacted content, mirroring `create-one.sh`'s exit-3 contract. Covered token families, explicit non-coverage, and PEM handling are inherited from `scripts/redact-secrets.sh`. Namespace note: the failure envelope is `FAILED=true` / `ERROR=<msg>` â€” NOT `ISSUE_FAILED=true` / `ISSUE_ERROR=<msg>` (the prefix used by `/issue`'s `create-one.sh`); this divergence is intentional and documented in `scripts/tracking-issue-write.md`. `scripts/test-tracking-issue-write.sh` is wired into `make lint` via `test-harnesses` and exercises eleven assertion categories (a)â€“(k) â€” see the ID table in `scripts/test-tracking-issue-write.md` for the canonical list â€” including gh-stderr redaction and the multiple-anchor fail-closed branch (exits 2 with `FAILED=true ERROR=multiple anchor comments found (ids: <list>)`). **Anchor body-level truncation posture** â€” `tracking-issue-write.sh` applies a body-level cap (`BODY_CAP=60000`) and per-section cap (`PER_SECTION_CAP=8000`) with deterministic marker-preserving collapse: the HTML anchor marker and every `<!-- section:<id> -->` / `<!-- section-end:<id> -->` marker remain intact through any collapse, so content loss via the collapse priority list is the worst outcome and the anchor skeleton cannot be corrupted or escaped. Redaction precedes truncation (see the Redaction order invariant above), so a truncated anchor does not imply secrets leaked â€” the scrubber has already run over the composed body before any bytes are sliced. This is a low-severity size-exhaustion / context-bloat mitigation, not a denial-of-service guarantee. See `scripts/tracking-issue-write.md` for the full collapse priority list.

Inline `[TRUNCATED â€” â€¦]` and `[section '<id>' truncated â€” â€¦]` markers produced by `tracking-issue-write.sh` are preserved verbatim in `TASK_FILE` â€” the read script's job is transmission, not reinterpretation.

### Automatic research-issue publication (`/research` Step 3.5)

By default, `/research` creates a GitHub issue at the end of each successful run containing the full research report and token spend metadata. This changes `/research` from producing a local/terminal-only artifact to publishing the full report to GitHub. The `--no-issue` flag suppresses this behavior.

**Redaction backstop**: `/issue`'s outbound shell scrubber (`scripts/redact-secrets.sh`) covers common secret patterns (API keys, tokens, passwords, certificates). It does NOT cover internal hostnames/URLs, PII, or domain-specific sensitive content.

**Residual risk**: research reports may contain security-sensitive findings (especially from `/research --scale=deep`'s `Codex-Sec` lane), internal architecture details, vulnerability assessments, or references to private infrastructure. Operators running `/research` against security-sensitive codebases should use `--no-issue` or review the generated issue after creation. The `--keep-sidecar` flag's existing guidance ("review and apply redaction if needed before filing") applies equally to the auto-filed issue â€” the auto-issue bypasses the manual review step that `--keep-sidecar` + `--dry-run` provides.

**Transitive callers**: `scripts/eval-research.sh` and `skills/skill-evolver/SKILL.md` pass `--no-issue` to suppress auto-issue when `/research` is invoked as an intermediate step rather than a user-facing research task.

## Fixed-string matching for interpolated values (issue #775 unified grep -F doctrine)

Any shell-script call site that matches a literal value (label name, comment marker, edge identifier) MUST use a fixed-string matcher â€” `grep -F` family or `awk` with `index()`/field-equality â€” when the searched-for value is interpolated from a variable. Using BRE/ERE matchers (`grep -E`, `grep` default) on interpolated values is a regex-injection class: a value containing regex metacharacters (`.`, `[]`, `(`, `^`, `$`, `|`, `*`, `+`, `?`, `{}`) can over-match, fail closed unexpectedly, or shift match semantics in ways the call site does not anticipate. The `-x` (whole-line) and `-q` (quiet) flags are independent and may be combined with `-F`.

**Sites currently following the doctrine** (closes #775; closes #743 as duplicate):

- **`skills/issue/scripts/create-one.sh:182`** â€” label-existence probe: `gh label list ... --jq '.[].name' | grep -Fqx -- "$L"`. Active shipped path: `/umbrella` forwards operator `--label` values verbatim through `/issue` to here. A label like `bug.feature` or `release[2026]` cannot false-match a regex-interpretation sibling.
- **`skills/umbrella/scripts/helpers.sh` wire-dag back-link probe** â€” anchored fixed-string via awk: `awk -v m="$backlink_marker" 'index($0, m) == 1 { found=1; exit } END { exit !found }'`. The `index() == 1` semantic preserves the line-start anchor that issue #716 review FINDING (Codex) deliberately added (defends against prose comments that quote the marker mid-line) while removing the regex interpretation of `${UMBRELLA}` and the rest of the marker.
- **`skills/umbrella/scripts/helpers.sh` wire-dag existing-edge probe** â€” fixed-string field-equality via awk: `awk -F$'\t' -v b="$blocker" -v t="$blocked" '($1 "") == b && ($2 "") == t {found=1; exit} END{exit !found}' "$EXISTING_EDGES_TSV"` (the empty-string concatenations force awk to compare as strings rather than auto-numeric). Closes the third regex-injection site on the wire-dag surface.
- **`skills/umbrella/scripts/helpers.sh` wire-dag `--umbrella` arg-parse boundary** â€” input-boundary numeric grammar guard: non-empty UMBRELLA values must match `^[1-9][0-9]*$`. The `[ -n "$UMBRELLA" ]` gate preserves the empty-string + `--no-backlinks` bypass path; non-empty non-numeric values fail fast with `ERROR=wire-dag --umbrella must be a positive integer`. Defense-in-depth: makes the input contract explicit so any future caller supplying a non-numeric umbrella value (a new orchestrator, a misconfigured forwarder, a mocked `gh api` response in a future test harness) fails at parse time rather than producing an invalid-ERE grep silently swallowed by surrounding `|| true`.

**Sites NOT a regex-injection concern** (regex pattern is a literal, only data is variable): static patterns like `^[1-9][0-9]*$` for input validation are fine â€” the regex is hardcoded in the script, only the data being matched is variable. Only call sites that interpolate variable values INTO the pattern are the doctrine's concern.

Edits to call sites that look like `grep -E "...${VAR}..."` or `grep "...${VAR}..."` MUST include a same-PR review for whether the value can carry regex metacharacters; if yes, switch to `grep -F`, `awk index()`, or `awk` field-equality, preserving any structural anchors (line-start, whole-line) via the chosen primitive's idiomatic equivalent. Test-coverage convention: pin the regex-metacharacter regression in the closest test harness so the doctrine becomes a CI-enforced invariant, not a one-off code-review.

# render-batch-input.sh — sibling contract

**Purpose**: convert an LLM-emitted `pieces.json` (an array of `{title, body, depends_on:[int,...]}`) into the markdown batch-input file consumed by `/issue --input-file`. Validates JSON shape (≥2 entries; each has non-empty `title` + `body`; each `depends_on` is an array of 1-based integers strictly less than the entry's index — i.e. only references earlier pieces).

**CLI**: `--tmpdir DIR --pieces-file FILE`. Both required; `--tmpdir` must exist AND be writable (separate guards — existence is checked first; writability is checked next so an unwritable directory rejects with `ERROR=tmpdir not writable: <path>` BEFORE any redirect under `$TMPDIR` produces a raw bash "Permission denied"). `--pieces-file` must be non-empty.

**Output**: writes `$TMPDIR/batch-input.md` (one `### <title>` block per piece followed by the body). Stdout grammar:
```
BATCH_INPUT_FILE=<absolute path>
PIECES_TOTAL=<N>
PIECE_<i>_TITLE=<title>           # i is 1-based, repeated for each piece
PIECE_<i>_DEPENDS_ON=<csv-of-ints> # may be empty
```

**Exit codes**: `0` success; `1` invalid input (`ERROR=…` on stderr).

**Dependencies**: `jq` is required (matches the rest of the larch toolchain — `/issue` already requires `jq`).

**Edit-in-sync rules**: any change to the `pieces.json` schema OR the markdown output shape OR stdout grammar requires a same-PR update to `SKILL.md` Step 3B.1 (which writes `pieces.json` and parses the stdout) and `/issue`'s batch-mode parser at `skills/issue/scripts/parse-input.sh` (which consumes the markdown).

**Test coverage**: the malformed-`pieces.json` gatekeeper contract is pinned by `test-render-batch-input.sh` (sibling `test-render-batch-input.md`), wired into `make lint` via the `test-umbrella-render-batch-input` Makefile target. The harness pins SIX failure modes at the script boundary: (a) JSON parse failure (unclosed brackets, garbage payloads) under the `ERROR=invalid pieces.json: <reason>` stderr prefix + exit 1 contract; (b) valid JSON whose top-level value is not an array (object root with ≥2 keys, string root, etc.) under the same `ERROR=invalid pieces.json:` prefix; (c) per-entry `depends_on` containing a non-integer number (e.g. `[1.5]`) under the `ERROR=pieces.json entry <i> has out-of-range depends_on values:` prefix + exit 1 — pinned by case 7, closes #647; (d) per-entry `title` containing an embedded LF under the `ERROR=pieces.json entry <i> title contains embedded newline` prefix + exit 1 — pinned by case 8, closes #648; (e) `--tmpdir` directory exists but is not writable, under the `ERROR=tmpdir not writable: <path>` prefix + exit 1 — pinned by the `assert_unwritable_tmpdir` case, closes #687; and (f) per-entry `body` containing a line that begins with `###` (three hashes + space, either at body start or after a `\n`) under the `ERROR=pieces.json entry <i> body contains line starting with '### '` prefix + exit 1 — pinned by the `assert_invalid_body` cases (mid-body and body-start positions), closes #847. Without the case-(d) guard, a multi-line `title` would later be emitted via `printf 'PIECE_<i>_TITLE=%s\n' "$title"` and split one logical KV into multiple physical stdout lines, silently breaking the one-KV-per-line grammar that `umbrella` SKILL.md Step 3B.1 parses. Without the case-(e) guard, an unwritable `--tmpdir` would surface as a raw bash "Permission denied" on the first redirect under `$TMPDIR` (`2>"$JQ_PARSE_ERR"` or `: > "$OUT"`) — same bug class as #645 on `render-umbrella-body.sh`. Without the case-(f) guard, a piece body containing a `###` line would flow verbatim into `batch-input.md` and be re-parsed by `/issue --input-file`'s line-based parser (`parse-input.sh` Path 3, generic mode) as a new-item boundary — silently splitting one piece into multiple parsed items with corrupted titles and broken `depends_on` index alignment, the producer-side counterpart to the consumer-side rule documented in `skills/issue/scripts/parse-input.md` "Reverse coupling: `/umbrella`'s piece bodies (#831)". The remaining per-entry shape validation (`title` / `body` non-empty) is covered by integration via `SKILL.md` end-to-end runs.

**Single-line stdout free-text fields (which are guarded, which are exempt)**: only `title` is emitted as a `KEY=value` stdout line that carries free-text from `pieces.json` (`PIECE_<i>_TITLE=<title>`); the case-(d) guard rejects embedded LF on this field. `body` is **not exempt** — although it flows into the markdown output file rather than stdout, the case-(f) guard rejects line-start `###` patterns to keep its content inert against `/issue --input-file`'s downstream line-based re-parse (`parse-input.sh` Path 3 generic-mode item-boundary detection); multi-line content remains allowed otherwise. `depends_on` is exempt — it is emitted as `PIECE_<i>_DEPENDS_ON=<csv>` but is built by `jq` from values already validated as integers in `[1, idx]` (per the `bad_deps` predicate), so structurally cannot contain newlines. `BATCH_INPUT_FILE` and `PIECES_TOTAL` are script-owned values, not `pieces.json`-derived. **Residual gap (title)**: the case-(d) guard rejects only LF (`\n`); CR-only (`\r`), Unicode line separators (U+2028 / U+2029), and other line-break code points are NOT rejected. Downstream consumers that treat stdout as strictly line-oriented should rely on the documented "no embedded LF" guarantee — broader sanitization is out of scope for #648. **Residual gap (body)**: the case-(f) guard matches only the literal pattern `###` (three hashes + a single space) at line start, since that is the producer of the silent-corruption bug at the documented `parse-input.sh` Path 3 trigger — full regex `^\#\#\#[[:space:]]+(.+)$` (i.e. start-of-line + three hashes + one or more whitespace chars + at least one non-newline content char + end-of-line; line `parse-input.sh:393`). The trailing `(.+)$` capture is what makes Path 3 strictly narrower than the case-(f) guard's literal four-char prefix match — see the Conservative-rejection note below for the bare-heading divergence. Variants where the hashes are separated from the heading text by tab, non-breaking space, or any other whitespace code point other than ASCII 0x20 are NOT rejected even though they would also trip the parser regex; fenced-code-block awareness is also intentionally out of scope per #847's "pick the simpler approach" directive. **Conservative-rejection note**: the guard is a strict-superset of `parse-input.sh` Path 3's trigger condition along TWO dimensions, both intentional collateral per #847's "pick the simpler approach" directive — do NOT "fix" the script to a more nuanced grammar that would miss either nuance.

1. **Bare-heading false positive**: Path 3 requires the post-whitespace `(.+)` capture to be non-empty (a title-bearing heading line), but the guard rejects on the literal four-char prefix alone — so a body line that is `###` followed by only whitespace, with no title text, is rejected here even though it would NOT split the piece downstream. Rare and ambiguous (likely an LLM-emission accident).
2. **`### OOS_<N>:` false positive**: `parse-input.sh` runs OOS-heading detection (regex `^\#\#\#[[:space:]]+OOS_[0-9]+:[[:space:]]+(.+)$`) BEFORE the plain `^\#\#\#[[:space:]]+(.+)$` branch. The OOS branch's #132 guard at `parse-input.sh:381-382` ABSORBS an `### OOS_<N>: …` line into an active generic body (`CURRENT_MODE=="generic"` AND `IN_BODY==true` AND non-empty `CURRENT_BODY`) instead of splitting. The case-(f) guard rejects bodies containing such lines anyway, since `### OOS_<N>: …` content inside an `/umbrella` piece body is itself anomalous (piece bodies are implementation contracts, not OOS-list scaffolding). Accepted trade-off: a parser-aware case-(f) guard that mirrored the OOS-then-plain ordering would couple `render-batch-input.sh` to multiple `parse-input.sh` regexes plus the pending-heading state — heavy for a renderer's input validator.

The simpler match grammar (`###` followed by a single ASCII space at line start) is preferred. Operators emitting genuinely benign body content with the same prefix (e.g., a markdown sample whose first non-fenced line happens to be a level-3 heading) should escape the line at composition time (e.g., `\\### …`, `​### …`) or restructure to `**bold**`/`####` per the producer-side rule in `/umbrella` SKILL.md Step 3B.1. The producer-side prompt directive in `/umbrella` SKILL.md Step 3B.1's `body` bullet remains the first-line rule; this guard is the mechanical backstop for the most common offending pattern.

**Piece-body item-split hazard (#831)**: each piece body in the rendered batch markdown is concatenated after a `### <piece title>` line and flows verbatim into `/issue --input-file`'s parser at `skills/issue/scripts/parse-input.sh`. That parser is line-based and, in **generic mode** (where `/umbrella`'s pieces always land — the surrounding `### <piece title>` heading itself triggers generic-mode entry via Path 3), treats any `^### <title>` line in a piece body as a new-item boundary (Path 3 at `parse-input.sh:393-423`: `flush_item` + start a new generic item). Result: the piece silently splits into multiple parsed items with wrong titles and broken `depends_on` index alignment relative to the original `pieces.json`. Piece bodies (the `body` field of each `pieces.json` entry rendered on this normal-mode pipeline) MUST NOT contain line-start `^### <title>` lines, including inside fenced code blocks (the parser does not track fence state). The producer-side rule is enforced at the prompt boundary in `skills/umbrella/SKILL.md` Step 3B.1's `body` bullet; the consumer-side catch-all (covering operator-prebuilt batches that bypass Step 3B.1 AND this renderer — `umbrella --input-file` / `issue --input-file` direct — plus other upstream producers) lives in `skills/issue/scripts/parse-input.md` under "Reverse coupling: `/umbrella`'s piece bodies (#831)". Scope of THIS contract is the **normal-mode pipeline only**: pre-decomposed-input mode skips both `/umbrella` Step 3B.1 and this renderer entirely, so the producer-side prompt directive does not run for those callers. There is no body-line escape mechanism in `render-batch-input.sh` analogous to `/research`'s `\\### …` defense (see `skills/research/scripts/render-findings-batch.md` "Body composition"); protection here is contract-level only.

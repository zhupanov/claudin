# skills/simplify-skill/scripts/build-feature-description.sh — contract

`skills/simplify-skill/scripts/build-feature-description.sh` resolves the target skill, enumerates in-scope `.md` files, and composes the `/implement` feature description for `/simplify-skill`. Required flag: `--name <skill-name>` (bare target skill name, `[a-z0-9-]+`, no colons). Stdout contract (always one `KEY=VALUE` per line) on success: `STATUS=ok`, `TARGET_SKILL_MD=<absolute-path-to-target-SKILL.md>`, `TARGET_DIR=<absolute-path-to-target-skill-dir>`, `INCLUDED_FILES=<abs-path-1>[:<abs-path-2>…]` (colon-separated, `SKILL.md` first, assumes no `:` in file paths), `FEATURE_FILE=<absolute-path-to-composed-feature-description>`; on failure: `STATUS=not_found|bad_name|…`. Fail-closed: exit non-zero on malformed input; emit a `STATUS=` line on every path so callers can parse it deterministically. The script's in-file header (lines 1–40) is the authoritative developer-facing spec for flag list, stdout KV grammar, enumeration scope, and fail-closed exit codes — read the header comment first when editing. This sibling `.md` exists so the co-location convention (see `AGENTS.md § Editing rules`) is uniformly applied across `skills/<name>/scripts/` — edits that change the stdout contract must update both the in-file header and this sibling in the same PR.

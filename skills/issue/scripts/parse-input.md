# skills/issue/scripts/parse-input.sh — contract

`skills/issue/scripts/parse-input.sh` parses `/issue` batch-mode input into per-item structured output. Requires `--input-file FILE` and `--output-dir DIR` (both mandatory — missing either fails fast with usage error on stderr). Bodies are written as plain text to `$OUTPUT_DIR/item-<i>-body.txt` (preserving newlines via `printf '%s'`), and stdout emits `ITEM_<i>_BODY_FILE=<absolute-path>` in place of the pre-#402 inline base64 line. Stdout carries only short structured KV lines (titles, reviewer/phase/vote-tally, MALFORMED flags, ITEMS_TOTAL) — no payload bodies, so no long opaque strings enter the caller's post-tool-use context. Non-zero exit means the parse failed (missing flags, missing input, or any file-write failure via `set -euo pipefail`); callers MUST treat captured stdout as unreliable on non-zero exit and skip downstream processing. `skills/issue/scripts/test-parse-input.sh` is the regression harness, wired into `make lint` via the `test-parse-input` target so parser regressions cannot ship undetected — it covers baseline cases, boundary cases, issues #129/#131/#132/#138, the negative-flag paths, and pins the "no `ITEM_<i>_BODY=` on stdout" invariant via a `grep -E` guard inside `run_parser`.

## Reverse coupling: `/research`'s sidecar shape (#510)

`skills/research/scripts/render-findings-batch.sh` emits a sidecar designed to round-trip cleanly through this parser's **generic-mode** path (CURRENT_MODE=generic — `### <title>` heading + free-form body, with no `### OOS_N:` items). The sidecar's metadata lines are intentionally formatted WITHOUT the leading <code>- </code> that would trigger this script's OOS field branches (`- **Description**:` / `- **Reviewer**:` / `- **Vote tally**:` / `- **Phase**:`); they appear as plain `**Source**:` / `**Risk**:` etc. lines and pass through as ordinary body content via the `IN_BODY=true` continuation branch.

**Edit-in-sync rule**: changes to this script's generic-mode behavior — particularly the `^\#\#\#[[:space:]]+(.+)$` regex (which decides what starts a new generic item) and the OOS field-branch gates that condition on `CURRENT_MODE=oos` — require re-running `/research`'s `make lint` even when only `skills/issue/` is edited. The cross-skill regression test is `skills/research/scripts/test-render-findings-batch.sh` (wired into `make lint` via `test-render-findings-batch`); it asserts `ITEMS_TOTAL` matches the helper's `COUNT` and that no items carry `MALFORMED=true`. The `render-findings-batch.sh` body-line escape (prefix `\` on any literal `### Foo` line in finding prose) defends against accidental item splits when a finding's prose contains a literal heading-shaped line — see `skills/research/scripts/render-findings-batch.md` "Body composition" for details.

## Reverse coupling: `/umbrella`'s piece bodies (#831)

`/umbrella` decomposes a task into work-pieces and emits each as a `### <piece title>` block plus piece body verbatim via `skills/umbrella/scripts/render-batch-input.sh`, then forwards the rendered batch markdown to `/issue --input-file`. Each piece body lands in this script's **generic-mode** path (CURRENT_MODE=generic — the surrounding `### <piece title>` heading itself drives generic-mode entry via Path 3 at the regex match below). Any further `^### <subheading>` line inside a piece body is then matched by the same Path 3 branch (`elif [[ "$line" =~ ^\#\#\#[[:space:]]+(.+)$ ]]` at lines 393-423: `flush_item` + start-new-generic-item), silently splitting the piece into multiple parsed items with wrong titles and broken `depends_on` index alignment relative to the original `pieces.json`.

This hazard applies to **every** upstream producer of `/issue --input-file` batch markdown:

- `/umbrella` normal-mode pipeline (Step 3B.1 → `render-batch-input.sh` → `/issue --input-file`).
- `/umbrella --input-file` (operator-prebuilt batches) — bypasses both Step 3B.1's prompt directive AND `render-batch-input.sh`, so the producer-side defenses in those surfaces do NOT run for these callers; this consumer-side subsection is the catch-all surface they need to consult.
- `/issue --input-file` direct invocations (operator authoring batch markdown by hand).

Unlike `/research` (which mitigates via a producer-side body-line backslash escape — prefix `\` on any literal `### Foo` line in finding prose; see `skills/research/scripts/render-findings-batch.md` "Body composition"), **`/umbrella` has no mechanical escape mechanism in `render-batch-input.sh`**: enforcement is documentation/contract-only. The rule is stated at three coordinated surfaces — `skills/umbrella/SKILL.md` Step 3B.1's `body` bullet (prompt-side directive that runs only on the normal-mode pipeline), `skills/umbrella/scripts/render-batch-input.md` "Piece-body item-split hazard" (producer-side contract scoped to the normal-mode pipeline), and this consumer-side catch-all subsection (covers all upstream producers including those that bypass the renderer).

**Edit-in-sync rule**: changes to this script's `^\#\#\#[[:space:]]+(.+)$` regex or Path 3 `flush_item` + start-new-generic-item semantics affect both reverse couplings (`/research` and `/umbrella`). The `/research` Edit-in-sync rule above already requires re-running `make lint` via `test-render-findings-batch`; `/umbrella`'s harness coverage for the SKILL.md prohibition is documented in `skills/umbrella/scripts/test-umbrella-emit-output-contract.md` (which deliberately does NOT pin the SKILL.md line-126 unbundled-`body` prohibition added in #831 — see that file's "Coverage asymmetry" subsection for rationale).

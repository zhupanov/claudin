# skills/issue/scripts/parse-input.sh — contract

`skills/issue/scripts/parse-input.sh` parses `/issue` batch-mode input into per-item structured output. Requires `--input-file FILE` and `--output-dir DIR` (both mandatory — missing either fails fast with usage error on stderr). Bodies are written as plain text to `$OUTPUT_DIR/item-<i>-body.txt` (preserving newlines via `printf '%s'`), and stdout emits `ITEM_<i>_BODY_FILE=<absolute-path>` in place of the pre-#402 inline base64 line. Stdout carries only short structured KV lines (titles, reviewer/phase/vote-tally, MALFORMED flags, ITEMS_TOTAL) — no payload bodies, so no long opaque strings enter the caller's post-tool-use context. Non-zero exit means the parse failed (missing flags, missing input, or any file-write failure via `set -euo pipefail`); callers MUST treat captured stdout as unreliable on non-zero exit and skip downstream processing. `skills/issue/scripts/test-parse-input.sh` is the regression harness, wired into `make lint` via the `test-parse-input` target so parser regressions cannot ship undetected — it covers baseline cases, boundary cases, issues #129/#131/#132/#138, the negative-flag paths, and pins the "no `ITEM_<i>_BODY=` on stdout" invariant via a `grep -E` guard inside `run_parser`.

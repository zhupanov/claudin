# scripts/lint-skill-invocations.py — contract

`scripts/lint-skill-invocations.py` is a guardrail lint (closes #159, #180) that enforces the sub-skill-invocation style guide (`skills/shared/subskill-invocation.md`) at two levels for SKILL.md files declaring `Skill` in `allowed-tools`. **Check 1 — total omission** (file-level): the body must contain at least one of the two canonical phrases — `Invoke the Skill tool` (Pattern A) or `via the Skill tool` (Pattern B). **Check 2 — per-invocation** (line-local, closes #180): for each body line that matches `INVOCATION_LINE_REGEX` — direct imperative `Invoke`/`re-invoke` (optionally `the` + a bounded `**bold-span**`) immediately followed by a backticked `` `/<name>` `` token, optionally followed by `skill` — the same line must also contain `via the Skill tool`. The regex is intentionally narrow: sub-procedure references like "Invoke the **Rebase + Re-bump Sub-procedure** ... `/bump-version`" and helper-script citations like "always invoke the helper script before calling `/bump-version`" are exempt because the slash command is not the immediate object of the verb. Lines inside fenced code blocks (delimited by ` ``` `) are exempt from the per-invocation check; the canonical phrases (`PATTERN_A_PHRASE`, `PATTERN_B_PHRASE`) and the regex (`INVOCATION_LINE_REGEX`) are declared as module constants at the top of the script — editing the canonical phrases in `skills/shared/subskill-invocation.md` requires updating these constants and re-running the harness in the same PR. Per-invocation messages emit absolute file line numbers (frontmatter offset preserved by `extract_frontmatter_and_body`'s third return value) so editor jump-to-line lands on the offending row. Both checks contribute to the exit-1 violation count; the file-level message uses the legacy "declares 'Skill' in allowed-tools but contains no ..." wording for grep parity, and per-invocation messages use the prefix "lint-skill-invocations: <path>:<line>: 'Invoke `/<cmd>`' without 'via the Skill tool' on the same line — see skills/shared/subskill-invocation.md". Exit-code semantics unchanged: `0` clean, `1` violations, `2` internal error (precedence preserved when both occur). The lint is registered as a pre-commit local hook (id `lint-skill-invocations`) with `additional_dependencies: ['pyyaml==6.0.2']` in `.pre-commit-config.yaml`, which installs PyYAML into the hook's isolated venv. The `test-lint-skill-invocations` Makefile target (a `make lint` prerequisite) also invokes `python3 scripts/lint-skill-invocations.py` directly outside the pre-commit venv, so CI's `ci.yaml` install step additionally installs `pyyaml==6.0.2` into the ambient Python so the harness can import it. The pyyaml version is pinned identically in both locations — if you bump one, bump the other. `scripts/test-lint-skill-invocations.sh` is its black-box regression harness (inline heredoc fixtures under `mktemp -d`, invokes the lint via `python3 scripts/lint-skill-invocations.py --root "$TMPROOT"`), wired into `make lint` via the `test-lint-skill-invocations` target; the harness is added to `agent-lint.toml`'s exclude list because agent-lint does not follow Makefile-only references. The `--root` flag is a contract token — it isolates fixtures during testing and must not be removed from the CLI.

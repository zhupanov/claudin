# scripts/collect-agent-results.sh — contract

`scripts/collect-agent-results.sh` collects, validates, and optionally retries external reviewer outputs from `scripts/run-external-agent.sh`. It wraps `scripts/wait-for-reviewers.sh` for sentinel polling, validates each output (non-empty + retry on empty via `.meta` files), optionally invokes `scripts/validate-research-output.sh` on each `STATUS=OK` entry under `--substantive-validation` (Phase 3 of umbrella #413, closes #416), and emits structured `STATUS=OK|TIMED_OUT|FAILED|EMPTY_OUTPUT|SENTINEL_TIMEOUT|NOT_SUBSTANTIVE` results plus per-tool health monotonicity. Used by `/research` (Step 1.4 + Step 2.4 — `skills/research/references/research-phase.md` + `validation-phase.md`), `/review`, `/design` Step 2a.3 + Step 3 + Step 2a.5 dialectic + judge phases, and `/implement` Step 5 quick-mode reviewer collection. Runs `set -uo pipefail` (line 57) but deliberately **omits `set -e`** so non-zero exits from reviewer subprocesses and the validator are informational rather than aborting. **Bash 3.2 portability invariant at the validator call site (line 405):** when a caller passes `--substantive-validation` without `--validation-mode` (e.g., `/research` Step 1.4), `VAL_ARGS=()` is empty; on bash 3.2 (macOS default `/bin/bash`) the bare `"${VAL_ARGS[@]}"` expansion under `set -u` raises `unbound variable` and silently misclassifies every reviewer file as `STATUS=NOT_SUBSTANTIVE`. The validator invocation MUST therefore use the `"${VAL_ARGS[@]+"${VAL_ARGS[@]}"}"` idiom (precedented at `scripts/create-pr.sh:105` — `${GH_DRAFT_ARGS[@]+"${GH_DRAFT_ARGS[@]}"}`); a one-line WHY-comment at the call site documents the rationale so future readers do not "simplify" it back. Issue #511 was a regression of this invariant; `scripts/test-collect-agent-bash32.sh` (wired into `make lint` via `test-harnesses`) is the regression harness — Case 1 statically pins the idiom at line 405 (Linux-CI backstop on every PR; bash 4.4+ does not exhibit the runtime hazard), and Cases 2/3 exercise the actual collector under any `/bin/bash` whose version is `< 4.4` (bash 3.x or bash 4.0-4.3 — both still vulnerable; bash 4.4 was the fix release) to verify both empty-`VAL_ARGS` and non-empty `--validation-mode` paths behave correctly. Edits to the substantive-validation loop (lines 372-412) MUST keep the safe-expansion idiom at the validator call site; if the call-site shape changes, update the harness's Case 1 regex AND this contract together. The script's per-tool health is monotonic (any failure permanently flips a tool to unhealthy for the rest of the session — a later success does NOT flip it back); `--write-health <path>` reads prior state and merges monotonically when the file already exists.

# AGENTS.md

This repository **is** the larch Claude Code plugin. Editing here modifies what ships to consumers. See `README.md` for installation, features, env vars, and the full skill catalog.

## Repository layout

Plugin ships the entire repo. **Runtime surface**: `skills/`, `agents/`, `hooks/`, `scripts/`, `.claude-plugin/`. Everything else is supplementary (docs, CI config, `.claude/skills/`, dev settings).

## Editing rules

- Use `/bump-version` to change `.claude-plugin/plugin.json` version — it owns that commit; `Bump version to X.Y.Z` is a reserved commit message.
- `skills/shared/reviewer-templates.md` is the canonical source for the Code Reviewer archetype. `agents/code-reviewer.md` is generated from it via `scripts/generate-code-reviewer-agent.sh` — do not hand-edit the agent file. Edit the template and run `bash scripts/generate-code-reviewer-agent.sh` to regenerate; the `agent-sync` CI job enforces that the committed agent file matches generator output.
- Always respect `scripts/block-submodule-edit.sh`. If a hook blocks a write, investigate and resolve the underlying issue. The guard ships via `hooks/hooks.json` only — `.claude/settings.json` no longer mirrors it, so contributors developing in this repo should load larch as a plugin (`claude --plugin-dir .` or the local marketplace) to pick up the guard.
- After any change, run `/relevant-checks`.

Edit patterns only after reading `SECURITY.md`'s outbound-redaction subsection.

- `skills/fix-issue/scripts/parse-prose-blockers.sh` is the prose-dependency parser invoked by `skills/fix-issue/scripts/fetch-eligible-issue.sh`'s `prose_open_blockers()` function. It extracts same-repo issue numbers from a single document's text (issue body, or one comment body at a time) matched against the conservative keyword set (`Depends on`, `Blocked by`, `Blocked on`, `Requires`, `Needs` + whitespace + `#N`). Emphasis wrappers (`*`, `_`) are normalized before matching so the motivating `**#150**` formatting is matched; link-target forms (`[#150](url)`) and cross-repo references remain NON-matches by construction. The parser is fail-open (empty stdout + exit 0 on any error), so correctness is guarded by `skills/fix-issue/scripts/test-parse-prose-blockers.sh` — its offline regression harness wired into `make lint` via the `test-parse-prose-blockers` target. - `scripts/deny-edit-write.sh` is the skill-scoped PreToolUse hook registered in `skills/research/SKILL.md`'s `hooks:` frontmatter for matcher `Edit|Write|NotebookEdit` — it permits the call only when the tool's target path resolves to an absolute path under canonical `/tmp`, and denies every other outcome (missing path, relative path, traversal outside `/tmp`, symlink cycle, canonicalization failure, `jq` runtime failure). The script reads stdin JSON and extracts the target via `jq -r '[.tool_input.file_path, .tool_input.notebook_path] | map(select(type == "string" and length > 0)) | .[0] // empty'` — `map(select(length > 0))` is required because `jq`'s `//` treats the empty string as present, so a naïve `// empty` would let an empty `file_path` shadow a valid `notebook_path`. NotebookEdit uses `notebook_path`; the length-aware fallback prevents fail-open on that shape. Path resolution mirrors `scripts/block-submodule-edit.sh`: bounded symlink walk (max depth 40; cycle → deny), nearest-existing-ancestor probe via `dirname`, then canonicalization via `cd … && pwd -P` to handle macOS `/tmp` → `/private/tmp` aliasing. The allowed root is pre-computed once as `$(cd /tmp && pwd -P)`; the decision is exact equality or `$ALLOWED_ROOT/` prefix. On allow the script emits empty stdout and exits 0; on deny it emits a fixed-literal `hookSpecificOutput` JSON envelope and exits 0. The deny envelope is composed only from fixed ASCII literals (single fixed reason string, no runtime interpolation into the deny JSON) and is byte-identical across three branches: the top-level `jq`-absent static `printf` path, the `block()` helper's `jq -cn` path, and `block()`'s inner `jq -cn || printf '<static>'` fallback (which protects against a `jq` runtime failure degrading to exit 0 with empty stdout = silent-allow). This mirrors the `jq || printf` precedent in `scripts/block-submodule-edit.sh:25-32`. The hook is the sole mechanical enforcer of the `/tmp`-only policy; `allowed-tools` in `skills/research/SKILL.md` declares the orchestrator's surface (including `Edit`, `Write`, `NotebookEdit`, and `Skill`) but does NOT confine writes to `/tmp` — see `SECURITY.md` for the full residual-risk framing (hook `permissionDecision` semantics may vary by Claude Code version; `Skill` is deliberately unscoped). The hook matcher MUST remain `Edit|Write|NotebookEdit` — adding `Skill` would deny `/issue` delegations, breaking the read-only-repo contract's child-invocation story. `scripts/test-deny-edit-write.sh` is the table-driven regression harness (repo-deny, `/tmp`-allow for both new and existing files, `/tmp/../etc/passwd`-deny, relative-deny, NotebookEdit `notebook_path` allow/deny, empty-`file_path`-with-valid-`notebook_path` allow (guards the jq `//` empty-string edge case), empty-path fail-closed deny, malformed-JSON deny, idempotency, `jq`-absent static fallback byte-identity), wired into `make lint` via the `test-deny-edit-write` target; the harness FAILs (exit 1) when its own `jq` is missing because the assertions need a JSON parser. Edits to the hook must stay in sync with the harness. The test script is added to `agent-lint.toml`'s exclude list because agent-lint's dead-script rule does not follow Makefile-only references.
- `scripts/verify-skill-called.sh` is a generic mechanical post-invocation verifier for `Skill` tool calls (closes #160). Three mutually-exclusive modes: `--sentinel-file <path>` (file exists, is a regular file, and is non-empty), `--stdout-line <regex> --stdout-file <path>` (captured stdout contains a line matching the POSIX ERE, matched via `LC_ALL=C grep -E -q -- …` to pin locale and block option injection; empty regex rejected as argument error; grep exit 2 is treated as an internal fault per the fail-closed contract), and `--commit-delta <expected> --before-count <N>` (commit count ahead of main increased by exactly the expected amount). Emits `VERIFIED=true|false` and `REASON=<token>` on stdout; exit 0 for pass/fail, exit 1 for argument errors and internal faults (fail-closed). Intended as a defense-in-depth gate for `Skill` calls whose child skills have no dedicated domain-specific verifier. Commit counting is **shared** with `scripts/check-bump-version.sh` via `scripts/lib-count-commits.sh`; `check-bump-version.sh` still owns the `--mode pre`/`--mode post` bump workflow (HAS_BUMP/VERIFIED/STATUS stdout contract — see the dedicated bullet below), while `verify-skill-called.sh` adds generic sentinel/stdout/commit-delta modes. In `/implement` Step 8, the sentinel-file check runs alongside `check-bump-version.sh --mode post` — the two are complementary (sentinel = "skill wrote its artifact"; post = "skill created a commit"). `scripts/lib-count-commits.sh` is a sourced-only shell library (no shebang, not invokable directly) that defines `count_commits` for both `scripts/check-bump-version.sh` and `scripts/verify-skill-called.sh`. It distinguishes `ok` / `missing_main_ref` / `git_error` via a file-based side channel (`COUNT_COMMITS_STATUS_FILE`), preserves the existing `WARN: check-bump-version.sh:` stderr prefix for log parity, and explicitly documents that `.claude/skills/bump-version/scripts/classify-bump.sh`'s merge-base logic is a different concept intentionally not migrated. Do NOT invoke `lib-count-commits.sh` directly — it is sourced by the two consumers above. `scripts/test-verify-skill-called.sh` is the regression harness covering all three modes plus the cwd-neutral source chain via `check-bump-version.sh`, wired into `make lint` via the `test-verify-skill-called` target. Both `scripts/lib-count-commits.sh` and `scripts/test-verify-skill-called.sh` are added to `agent-lint.toml`'s exclude list because agent-lint does not follow `source` directives or Makefile-only references.
- `scripts/check-bump-version.sh` is the `/implement` Step 8 bump-workflow probe. Two modes: `--mode pre` emits `HAS_BUMP=true|false`, `COMMITS_BEFORE=<N>`, `STATUS=ok|missing_main_ref|git_error`; `--mode post --before-count <N>` emits `VERIFIED=true|false`, `COMMITS_AFTER=<N>`, `EXPECTED=<N>`, `STATUS=ok|missing_main_ref|git_error`. The `STATUS=` line plumbs the `COUNT_COMMITS_STATUS_FILE` side channel from `lib-count-commits.sh` so callers can distinguish a transient git error (corrupted pack, shallow-clone object boundary, PATH shim, etc.) from a legitimate "0 commits ahead" result (closes #172). `--mode post` forces `VERIFIED=false` whenever `STATUS != ok` — the numeric commit-delta is never trusted under degraded git state — which is the fail-closed invariant that fixes the historical silent-zero false-pass bug. Any unknown or empty token received from the side channel is normalized to `STATUS=git_error` (mirrors `verify-skill-called.sh`'s default branch that maps unknown to `REASON=git_error`). `scripts/test-check-bump-version.sh` is its regression harness covering all three status paths × both modes plus the `origin/main`-only fallback and the #172 fail-closed regression guard, wired into `make lint` via the `test-check-bump-version` target. The test script is added to `agent-lint.toml`'s exclude list because agent-lint does not follow Makefile-only references. Edits to the script must stay in sync with the harness.

- `scripts/lint-skill-invocations.py` is a guardrail lint (closes #159, #180) that enforces the sub-skill-invocation style guide (`skills/shared/subskill-invocation.md`) at two levels for SKILL.md files declaring `Skill` in `allowed-tools`. **Check 1 — total omission** (file-level): the body must contain at least one of the two canonical phrases — `Invoke the Skill tool` (Pattern A) or `via the Skill tool` (Pattern B). **Check 2 — per-invocation** (line-local, closes #180): for each body line that matches `INVOCATION_LINE_REGEX` — direct imperative `Invoke`/`re-invoke` (optionally `the` + a bounded `**bold-span**`) immediately followed by a backticked `` `/<name>` `` token, optionally followed by `skill` — the same line must also contain `via the Skill tool`. Lines inside fenced code blocks (delimited by ` ``` `) are exempt from the per-invocation check; the canonical phrases (`PATTERN_A_PHRASE`, `PATTERN_B_PHRASE`) and the regex (`INVOCATION_LINE_REGEX`) are declared as module constants at the top of the script — editing the canonical phrases in `skills/shared/subskill-invocation.md` requires updating these constants and re-running the harness in the same PR. Exit-code semantics unchanged: `0` clean, `1` violations, `2` internal error (precedence preserved when both occur). The lint is registered as a pre-commit local hook (id `lint-skill-invocations`) with `additional_dependencies: ['pyyaml==6.0.2']` in `.pre-commit-config.yaml`, which installs PyYAML into the hook's isolated venv. The pyyaml version is pinned identically in both locations — if you bump one, bump the other. `scripts/test-lint-skill-invocations.sh` is its black-box regression harness (inline heredoc fixtures under `mktemp -d`, invokes the lint via `python3 scripts/lint-skill-invocations.py --root "$TMPROOT"`), wired into `make lint` via the `test-lint-skill-invocations` target; the harness is added to `agent-lint.toml`'s exclude list because agent-lint does not follow Makefile-only references. ## Common editing tasks

- **Changing a skill** → start at `skills/<name>/SKILL.md`, then trace every helper in `skills/<name>/scripts/`, `scripts/`, and `skills/shared/`. Behavior is split between prompt and scripts.
- **Adding/modifying the Code Reviewer archetype** → edit `skills/shared/reviewer-templates.md` (canonical), then run `bash scripts/generate-code-reviewer-agent.sh` to regenerate `agents/code-reviewer.md`. For any other reviewer archetype, follow the general rule: identify the canonical source and mirror updates to any generated outputs.
- **Changing a shared script** → edit `scripts/<name>.sh`, then grep for callers across `skills/`, `hooks/`, `.claude/settings.json`, `.github/workflows/`, and other scripts.
- **Changing dev-only skills** → edit under `.claude/skills/bump-version/` or `.claude/skills/relevant-checks/`.
- **Docs or scripts only** → classified as PATCH.
- **Dialectic smoke test** → `scripts/dialectic-smoke-test.sh` is an offline fixture-driven regression guard for `/design` Step 2a.5. Fixtures live under `tests/fixtures/dialectic/` (a non-runtime path — everything outside the named runtime surface is supplementary). Run locally with `bash scripts/dialectic-smoke-test.sh` or via `make smoke-dialectic`; CI runs it in the `smoke-dialectic` job. When changing `skills/shared/dialectic-protocol.md` Parser tolerance or Threshold Rules sections, update the smoke test and/or fixtures in the same PR.

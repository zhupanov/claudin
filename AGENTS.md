# AGENTS.md

This repository **is** the larch Claude Code plugin. Editing here modifies what ships to consumers. See `README.md` for installation, features, env vars, and the full skill catalog.

## Repository layout

Plugin ships the entire repo. **Runtime surface**: `skills/`, `agents/`, `hooks/`, `scripts/`, `.claude-plugin/`. Everything else is supplementary (docs, CI config, `.claude/skills/`, dev settings).

## Editing rules

- Use `/bump-version` to change `.claude-plugin/plugin.json` version — it owns that commit; `Bump version to X.Y.Z` is a reserved commit message.
- `skills/shared/reviewer-templates.md` is the canonical source for the Code Reviewer archetype. `agents/code-reviewer.md` is generated from it via `scripts/generate-code-reviewer-agent.sh` — do not hand-edit the agent file. Edit the template and run `bash scripts/generate-code-reviewer-agent.sh` to regenerate; the `agent-sync` CI job enforces that the committed agent file matches generator output.
- Always respect `scripts/block-submodule-edit.sh`. If a hook blocks a write, investigate and resolve the underlying issue.
- After any change, run `/relevant-checks`.
- `scripts/redact-secrets.sh` is the outbound secret-scrubbing filter invoked by `skills/issue/scripts/create-one.sh` before `gh issue create`. `scripts/test-redact-secrets.sh` is its regression test, wired into `make lint` via the `test-redact` target. Edit patterns only after reading `SECURITY.md`'s outbound-redaction subsection.
- `skills/issue/scripts/parse-input.sh` parses `/issue` batch-mode input. `skills/issue/scripts/test-parse-input.sh` is its regression harness, wired into `make lint` via the `test-parse-input` target so parser regressions cannot ship undetected.
- `${CLAUDE_PLUGIN_ROOT}/scripts/sessionstart-health.sh` is the SessionStart preflight hook that probes `jq` and `git` on `PATH` at session start and injects an advisory into session context when either is missing. `${CLAUDE_PLUGIN_ROOT}/scripts/test-sessionstart-health.sh` is its regression test, wired into `make lint` via the `test-sessionstart` target (run manually via `bash ${CLAUDE_PLUGIN_ROOT}/scripts/test-sessionstart-health.sh`). The hook MUST always exit 0 (SessionStart is non-blocking by spec) and the `additionalContext` string MUST remain fixed ASCII literals — see the invariant comment in the script.
- `scripts/audit-edit-write.sh` is a dev-only PostToolUse audit hook for `Edit`/`Write` tool use. It is shipped in the plugin install tree but is **not registered by default** in `hooks/hooks.json` or `.claude/settings.json`; contributors opt in locally by adding a `PostToolUse` entry to `.claude/settings.local.json` (gitignored). Appends one JSONL record per invocation to `.claude/hook-audit.log` (also gitignored). `scripts/test-audit-edit-write.sh` is its regression harness, wired into `make lint` via the `test-audit-edit-write` target. See `docs/dev-hook-audit.md` for enable/rotate/privacy details and `SECURITY.md` for the security posture.
- `scripts/block-submodule-edit.sh` is the PreToolUse hook that denies edits to files inside submodules. `scripts/test-block-submodule-edit.sh` is its regression harness, wired into `make lint` via the `test-block-submodule` target; edits to the hook must stay in sync with the harness.
- `skills/create-skill/scripts/post-scaffold-hints.sh` is the post-scaffold reminder emitter invoked by `/create-skill`. It emits dual-form Skill permission hints (`Skill(<NAME>)` and `Skill(larch:<NAME>)`) in plugin-dev mode and references the README "Strict-permissions consumers — Skill permission entries" subsection for rationale. `scripts/test-post-scaffold-hints.sh` is its regression harness, wired into `make lint` via the `test-post-scaffold-hints` target; edits to the hints script must stay in sync with the harness. The dual-form convention and README cross-reference are contract tokens asserted by the harness — do not silently drop the `Skill(larch:$NAME)` line, the `sort -u` instruction, or the README-subsection pointer.
- `scripts/deny-edit-write.sh` is the skill-scoped PreToolUse deny hook registered in `skills/research/SKILL.md`'s `hooks:` frontmatter — it denies `Edit|Write|NotebookEdit` while `/research` is active by emitting a fixed `hookSpecificOutput` JSON deny envelope. The script ignores stdin (the matcher already restricts the trigger), is composed only of fixed ASCII literals (no runtime interpolation into the deny JSON), and always exits 0 — when `jq` is present it composes the JSON via `jq -cn`; when `jq` is absent it falls back to a byte-identical static `printf`, so the deny semantics never depend on `jq` availability. This mirrors the `jq`-absent precedent in `scripts/block-submodule-edit.sh`. This is a defense-in-depth second layer; the primary mechanical control remains the `allowed-tools` frontmatter that omits `Edit`/`Write`/`Skill` (see `SECURITY.md`). `scripts/test-deny-edit-write.sh` is its regression harness, wired into `make lint` via the `test-deny-edit-write` target; the harness FAILs (exit 1) when its own `jq` is missing because the assertions need a JSON parser, and it exercises both the `jq` and `printf` fallback branches to enforce the byte-identity invariant. Edits to the hook must stay in sync with the harness. The test script is added to `agent-lint.toml`'s exclude list because agent-lint's dead-script rule does not follow Makefile-only references.
- `scripts/verify-skill-called.sh` is a generic mechanical post-invocation verifier for `Skill` tool calls (closes #160). Three mutually-exclusive modes: `--sentinel-file <path>` (file exists, is a regular file, and is non-empty), `--stdout-line <regex> --stdout-file <path>` (captured stdout contains a line matching the POSIX ERE, matched via `LC_ALL=C grep -E -q -- …` to pin locale and block option injection; empty regex rejected as argument error; grep exit 2 is treated as an internal fault per the fail-closed contract), and `--commit-delta <expected> --before-count <N>` (commit count ahead of main increased by exactly the expected amount). Emits `VERIFIED=true|false` and `REASON=<token>` on stdout; exit 0 for pass/fail, exit 1 for argument errors and internal faults (fail-closed). Intended as a defense-in-depth gate for `Skill` calls whose child skills have no dedicated domain-specific verifier. Commit counting is **shared** with `scripts/check-bump-version.sh` via `scripts/lib-count-commits.sh`; `check-bump-version.sh` still owns the `--mode pre`/`--mode post` bump workflow (HAS_BUMP/VERIFIED/STATUS stdout contract — see the dedicated bullet below), while `verify-skill-called.sh` adds generic sentinel/stdout/commit-delta modes. In `/implement` Step 8, the sentinel-file check runs alongside `check-bump-version.sh --mode post` — the two are complementary (sentinel = "skill wrote its artifact"; post = "skill created a commit"). `scripts/lib-count-commits.sh` is a sourced-only shell library (no shebang, not invokable directly) that defines `count_commits` for both `scripts/check-bump-version.sh` and `scripts/verify-skill-called.sh`. It distinguishes `ok` / `missing_main_ref` / `git_error` via a file-based side channel (`COUNT_COMMITS_STATUS_FILE`), preserves the existing `WARN: check-bump-version.sh:` stderr prefix for log parity, and explicitly documents that `.claude/skills/bump-version/scripts/classify-bump.sh`'s merge-base logic is a different concept intentionally not migrated. Do NOT invoke `lib-count-commits.sh` directly — it is sourced by the two consumers above. `scripts/test-verify-skill-called.sh` is the regression harness covering all three modes plus the cwd-neutral source chain via `check-bump-version.sh`, wired into `make lint` via the `test-verify-skill-called` target. Both `scripts/lib-count-commits.sh` and `scripts/test-verify-skill-called.sh` are added to `agent-lint.toml`'s exclude list because agent-lint does not follow `source` directives or Makefile-only references.
- `scripts/check-bump-version.sh` is the `/implement` Step 8 bump-workflow probe. Two modes: `--mode pre` emits `HAS_BUMP=true|false`, `COMMITS_BEFORE=<N>`, `STATUS=ok|missing_main_ref|git_error`; `--mode post --before-count <N>` emits `VERIFIED=true|false`, `COMMITS_AFTER=<N>`, `EXPECTED=<N>`, `STATUS=ok|missing_main_ref|git_error`. The `STATUS=` line plumbs the `COUNT_COMMITS_STATUS_FILE` side channel from `lib-count-commits.sh` so callers can distinguish a transient git error (corrupted pack, shallow-clone object boundary, PATH shim, etc.) from a legitimate "0 commits ahead" result (closes #172). `--mode post` forces `VERIFIED=false` whenever `STATUS != ok` — the numeric commit-delta is never trusted under degraded git state — which is the fail-closed invariant that fixes the historical silent-zero false-pass bug. Any unknown or empty token received from the side channel is normalized to `STATUS=git_error` (mirrors `verify-skill-called.sh`'s default branch that maps unknown to `REASON=git_error`). `scripts/test-check-bump-version.sh` is its regression harness covering all three status paths × both modes plus the `origin/main`-only fallback and the #172 fail-closed regression guard, wired into `make lint` via the `test-check-bump-version` target. The test script is added to `agent-lint.toml`'s exclude list because agent-lint does not follow Makefile-only references. Edits to the script must stay in sync with the harness.
- Public `skills/*/SKILL.md` use `${CLAUDE_PLUGIN_ROOT}/…`; dev-only `.claude/skills/*/SKILL.md` use `$PWD/…`.
- `skills/shared/subskill-invocation.md` is the canonical source for sub-skill invocation conventions (Pattern A bulleted vs Pattern B inline, `allowed-tools` narrowing heuristic, post-invocation verification for orchestrators, `session-env` handoff and safe-parse rule, anti-conditional-phrasing for Skill-tool calls, bare-name-then-fully-qualified fallback). Runtime surface (ships to consumers under `skills/`). No generated artifact — update directly. Update trigger: when a cited source-example skill (`/im`, `/imaq`, `/alias`, `/create-skill`, `/fix-issue`, `/implement`) changes its invocation pattern, update the corresponding example in the guide in the same PR. `skills/create-skill/scripts/render-skill-md.sh` emits a `## Sub-skill Invocation` reminder block referencing this file into every scaffolded skill; `skills/create-skill/scripts/test-render-skill-md.sh` is the regression harness guarding that emission (wired into `make lint` via the `test-render-skill` target).
- `scripts/lint-skill-invocations.py` is a minimal-guardrail lint (closes #159) that flags SKILL.md files which declare `Skill` in `allowed-tools` but contain neither of the two canonical invocation phrases from `skills/shared/subskill-invocation.md` — `Invoke the Skill tool` (Pattern A) and `via the Skill tool` (Pattern B) — anywhere in the body. The lint catches *total omission* only; per-invocation alignment (every Skill-tool call at its exact site using Pattern A/B) is intentionally out of scope for this first-pass check and is tracked as a follow-up OOS issue. The two phrases are declared as `PATTERN_A_PHRASE` / `PATTERN_B_PHRASE` module constants at the top of the script; editing the canonical phrases in `skills/shared/subskill-invocation.md` requires updating the constants and re-running the harness in the same PR. The lint is registered as a pre-commit local hook (id `lint-skill-invocations`) with `additional_dependencies: ['pyyaml==6.0.2']` in `.pre-commit-config.yaml`, which installs PyYAML into the hook's isolated venv. The `test-lint-skill-invocations` Makefile target (a `make lint` prerequisite) also invokes `python3 scripts/lint-skill-invocations.py` directly outside the pre-commit venv, so CI's `ci.yaml` install step additionally installs `pyyaml==6.0.2` into the ambient Python so the harness can import it. The pyyaml version is pinned identically in both locations — if you bump one, bump the other. `scripts/test-lint-skill-invocations.sh` is its black-box regression harness (inline heredoc fixtures under `mktemp -d`, invokes the lint via `python3 scripts/lint-skill-invocations.py --root "$TMPROOT"`), wired into `make lint` via the `test-lint-skill-invocations` target; the harness is added to `agent-lint.toml`'s exclude list because agent-lint does not follow Makefile-only references. The `--root` flag is a contract token — it isolates fixtures during testing and must not be removed from the CLI.
- Update `SECURITY.md` when security-relevant behavior changes.

## Common editing tasks

- **Changing a skill** → start at `skills/<name>/SKILL.md`, then trace every helper in `skills/<name>/scripts/`, `scripts/`, and `skills/shared/`. Behavior is split between prompt and scripts.
- **Adding/modifying the Code Reviewer archetype** → edit `skills/shared/reviewer-templates.md` (canonical), then run `bash scripts/generate-code-reviewer-agent.sh` to regenerate `agents/code-reviewer.md`. For any other reviewer archetype, follow the general rule: identify the canonical source and mirror updates to any generated outputs.
- **Changing a shared script** → edit `scripts/<name>.sh`, then grep for callers across `skills/`, `hooks/`, `.claude/settings.json`, `.github/workflows/`, and other scripts.
- **Changing dev-only skills** → edit under `.claude/skills/bump-version/` or `.claude/skills/relevant-checks/`.
- **Docs or scripts only** → classified as PATCH.
- **Dialectic smoke test** → `scripts/dialectic-smoke-test.sh` is an offline fixture-driven regression guard for `/design` Step 2a.5. Fixtures live under `tests/fixtures/dialectic/` (a non-runtime path — everything outside the named runtime surface is supplementary). Run locally with `bash scripts/dialectic-smoke-test.sh` or via `make smoke-dialectic`; CI runs it in the `smoke-dialectic` job. When changing `skills/shared/dialectic-protocol.md` Parser tolerance or Threshold Rules sections, update the smoke test and/or fixtures in the same PR.

## Conventions

- Shell scripts use `set -euo pipefail` by default. Comment when `-e` is intentionally omitted.
- Follow recent commit history style. `Bump version to X.Y.Z` is reserved for `/bump-version`.
- Run `gh pr create` through the skill, not manually.
- Slack env vars are optional; skills degrade gracefully when absent.

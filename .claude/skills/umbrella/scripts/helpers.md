# helpers.sh тАФ sibling contract

Consolidated `/umbrella` helpers, exposed as three subcommands.

## `check-cycle --existing-edges FILE --candidate BLOCKER:BLOCKED`

Pure-logic DAG cycle test. `FILE` is a TSV of `<blocker>\t<blocked>` edges. The candidate adds `BLOCKER -> BLOCKED`. Stdout: `CYCLE=true|false`. Self-loops (`BLOCKER == BLOCKED`) are always cycles. The forward-reachability check from the new BLOCKED node looks for the new BLOCKER as an ancestor; reaching it means the new edge would close a cycle. Independently testable via `test-helpers.sh`.

## `wire-dag --tmpdir DIR --umbrella N --umbrella-title T --children-file F --edges-file E --repo R [--dry-run] [--no-backlinks]`

Coordinator: feature-detect the GitHub blocked-by dependency API (probe `/repos/<repo>/issues/<umbrella>/dependencies/blocked_by`), build `EXISTING_EDGES_TSV` from the full reachable `blocked_by` subgraph (transitive closure from children + both endpoints of every proposed edge тАФ issue #718), cycle-check each proposed edge, add survivors, then post back-link comments on each child unless the umbrella is already in the child's blocked_by list (native umbrella relationship).

The transitive `blocked_by` traversal is bounded by `WIRE_DAG_TRAVERSAL_NODE_CAP` (default 200, env override). Per-node `blocked_by` lookups are cached for the run (parallel to `BLOCKER_ID_CACHE`). The cycle-check graph captures cycles closing through non-child intermediaries тАФ pre-#718 the TSV was seeded only from each child's `blocked_by`, missing such cycles silently.

`--no-backlinks` (created-eq-1 bypass): when set, the umbrella-rooted API probe is replaced by a probe of the FIRST CHILD in `CHILDREN_FILE` (children always exist on this path; `--umbrella` may be empty). The entire back-link emission loop is skipped тАФ no native-relationship `gh api` calls, no `gh issue comment` posts, no use of `$UMBRELLA` / `$UMBRELLA_TITLE`. Edge wiring + cycle-check + idempotency + per-edge counters remain unchanged. Output keys `BACKLINKS_POSTED` and `BACKLINKS_SKIPPED_NATIVE` are still emitted on stdout (both will be `0` when the loop is skipped) but the orchestrator does NOT propagate them to `output.kv` on the bypass path (see Edit-in-sync rules below).

Per-edge POST body uses the canonical `{"issue_id": <internal numeric id>}` shape matching `skills/issue/scripts/add-blocked-by.sh:170-183`. Blocker internal ids are resolved via `gh api /repos/<repo>/issues/<N> --jq .id` and cached per run. Per-node `blocked_by` lookups go through `gh api "/repos/<repo>/issues/<N>/dependencies/blocked_by" --paginate --jq '.[].number'` (path before `--paginate`; matches the harness's `"$1 $2"` matcher and the wider repo `--paginate` convention in `skills/fix-issue/scripts/umbrella-handler.sh`). The back-link "umbrella already in child's `blocked_by`" probe uses the same endpoint with `--paginate --jq ".[] | select(.number == ${UMBRELLA})"` (issue #731): without `--paginate`, a child whose blocker list spans more than one page (>30 by GitHub's default page size) could have the umbrella appear on a later page, causing the script to either skip the back-link comment or post a redundant one.

Stdout grammar: `EDGES_ADDED=<N>`, per-edge `EDGE_<j>_BLOCKER=<N>` / `EDGE_<j>_BLOCKED=<M>`, `EDGES_REJECTED_CYCLE=<N>`, `EDGES_SKIPPED_EXISTING=<N>` (already-present edges, including idempotent 422 already-exists responses per `add-blocked-by.sh:193-196`), `EDGES_SKIPPED_API_UNAVAILABLE=<N>` (repo-wide probe failure OR per-edge feature-missing 404 тАФ both mean "the GitHub dependency surface was not usable here for this run"; **semantics intentionally preserved** by issue #728's fix тАФ see `PROBE_FAILED` below for the cause disambiguator), `EDGES_FAILED=<N>` covers two categories: (a) per-edge **operational** failures тАФ rate-limit, permission denied, ambiguous 404, other 4xx, 5xx, request-shape mismatches, blocker-id lookup failure, network тАФ failures that do NOT match the feature-missing fingerprint; (b) per-edge **policy-driven** failures тАФ `bound-exhausted`, set when the transitive `blocked_by` traversal hit `WIRE_DAG_TRAVERSAL_NODE_CAP` and the per-edge cycle-check loop's `CYCLE=false` answer cannot be trusted on a known-incomplete TSV (issue #718 fail-closed posture, voted 3-0 in the design-phase dialectic). `PROBE_FAILED=<0|1>` (issue #728) is a parse-only per-run flag distinguishing the two repo-wide-skip causes: `0` = confirmed feature-missing (probe got a fingerprinted 404) **OR** no probe attempted (e.g., `--no-backlinks` first-child path with empty `CHILDREN_FILE` тАФ `probe_target` was empty, no probe ran, `PROBE_FAILED` stays at its initialization value); `1` = transient/operational probe failure (5xx + retry also failed, or HTTP response other than 2xx/fingerprinted-404). The retry policy is one network-only retry on `5xx` and empty-status (no HTTP response at all); `429`, other `4xx`, and `2xx`/fingerprinted-404 are single-attempt (clear HTTP responses are not transport blips). `BACKLINKS_POSTED=<N>`, `BACKLINKS_SKIPPED_NATIVE=<N>`.

Stderr: five distinct warning prefixes:
1. **One warning** when the API surface is unavailable repo-wide AND `PROBE_FAILED=0` (confirmed feature-missing тАФ issue #728 gates this on `PROBE_FAILED=0` so the new probe-failed warning at prefix #5 is the only signal on the transient path; emitting both would double-warn). The warning text is mode-aware: in default mode (no `--no-backlinks`) the tail reads `Back-links posted via comments.` (back-links via plain comments still run as a fail-open fallback). In `--no-backlinks` mode (created-eq-1 bypass) the tail reads `Back-links suppressed (--no-backlinks).` so the message stays factually accurate on the bypass path: `**тЪа /umbrella: GitHub blocked-by dependency API not available on REPO; skipped DAG wiring. <Back-links posted via comments. | Back-links suppressed (--no-backlinks).>`
2. **One redacted single-line warning per `EDGES_FAILED` event** of the form `**тЪа /umbrella: wire-dag edge BLOCKER->BLOCKED failed (HTTP STATUS): REASON**`. Used for both operational (HTTP 4xx/5xx, id-lookup) and policy-driven (`bound-exhausted`) failures тАФ the `STATUS` placeholder carries the typed reason. The "HTTP" prose is a soft misnomer for the policy-driven case; preserved verbatim so the existing per-edge stderr count assertion (`grep -c 'тЪа /umbrella: wire-dag edge'`) remains stable. The reason snippet is piped through `scripts/redact-secrets.sh` (canonical secret scrubber) when present; a degraded-layout fallback flattens to `tr | head -c 200` and emits a one-time process-local notice.
3. **One-time-per-run** warning when the transitive traversal hit `WIRE_DAG_TRAVERSAL_NODE_CAP`: `**тЪа /umbrella: wire-dag traversal cap reached (cap=N, queue=M, elapsed=Ts) тАФ pending candidates will fail closed**`. This warning fires once at cap-hit; the per-candidate `EDGES_FAILED` warnings (prefix #2) follow for each affected edge.
4. **One-time-per-failed-node** warning for transient `blocked_by` lookup failures: `**тЪа /umbrella: wire-dag blocked_by lookup failed for #N тАФ treating as no edges**`. Individual lookup failures are fail-open (residual posture matching the file's existing API-blip handling; design-phase FINDING_5 was exonerated 2-1 тАФ the residual risk is that the failed-lookup node could be the sole cycle witness, which would produce a missed cycle on a single proposed edge). The `EDGES_FAILED` counter does NOT increment for lookup failures.
5. **One-time-per-run** warning for transient/operational repo-wide probe failure (issue #728): `**тЪа /umbrella: wire-dag probe failed (HTTP STATUS): REASON**`. Fires when the probe classified `PROBE_FAILED=1` тАФ i.e., 5xx after retry, no-status after retry, 429, or other non-fingerprinted 4xx. Mirrors prefix #2's redact-secrets fail-closed discipline (substitutes `<REDACTION_FAILED>` placeholder when `redact-secrets.sh` exits non-zero; never prints the raw flattened body). One probe per run, so this warning fires at most once. When `PROBE_FAILED=1`, prefix #1 (the legacy "API not available" warning) does NOT fire тАФ only the new prefix #5 emits тАФ to avoid double-warning on the same condition. The shared `REDACT_FALLBACK_WARNED` one-time guard is intentionally per-run (layout-context, not call-site-context): if the probe path trips the fallback notice first, later per-edge failures inherit the same notice without re-emitting it.

**Residual redaction risk**: `scripts/redact-secrets.sh` covers token-shaped secrets (API keys, JWTs, PEM blocks, common provider key prefixes) but does NOT cover PII, internal hostnames, opaque bearer tokens, or DB connection strings (see the script's header for the explicit non-coverage list). Operators should treat these warnings like any other internal log line тАФ best-effort secret scrubbing is not comprehensive PII protection.

**Semantic migration note (issue #720)**: Before this change, post-probe per-edge POST failures (HTTP 429 rate-limit, 403 permission denied, ambiguous 404, 5xx, request-shape mismatches) silently incremented `EDGES_SKIPPED_API_UNAVAILABLE` and emitted no diagnostic. They now increment `EDGES_FAILED` and emit one redacted stderr warning each. Dashboards or playbooks keyed on the old counter as a "benign skip volume" gauge will see a drop in `EDGES_SKIPPED_API_UNAVAILABLE` matched by a corresponding rise in `EDGES_FAILED`, plus new stderr noise тАФ both intentional. The previous broken POST body shape (`-f issue_number=<display>`) was simultaneously fixed to the canonical `issue_id` form, so `EDGES_ADDED` should also see a corresponding rise on repos where the dependencies feature is enabled.

## `emit-output --kv-file FILE`

Validate the LLM-supplied `output.kv` (no malformed lines, no duplicate keys) and stream it to stdout. Defense-in-depth on top of the `SKILL.md` Step 4 grammar. For `emit-output` specifically, stderr is reserved for parse/validation/usage errors only тАФ the human summary breadcrumb is emitted by the orchestrator at SKILL.md Step 4 (single emission point), not by this script. (This scoping is local to `emit-output`; `wire-dag`'s documented stderr warning behavior above is unaffected.)

### Edit-in-sync rules

Changes to `wire-dag` stdout keys consumed by orchestrator emit-output (i.e., keys that flow through `output.kv` per SKILL.md Step 4) require a same-PR update to `SKILL.md` Step 4 AND Step 3B.4. New `wire-dag` stdout keys consumed only for orchestrator parsing / session state (NOT propagated through `output.kv`) require updates only to `SKILL.md` Step 3B.4 + this file's stdout-grammar paragraph above. Today the `output.kv`-propagated set is `EDGES_ADDED`, `EDGE_<j>_BLOCKER`, `EDGE_<j>_BLOCKED`, `BACKLINKS_POSTED`; the parse-only set is `EDGES_REJECTED_CYCLE`, `EDGES_SKIPPED_EXISTING`, `EDGES_SKIPPED_API_UNAVAILABLE`, `EDGES_FAILED`, `PROBE_FAILED`, `BACKLINKS_SKIPPED_NATIVE`.

**`--no-backlinks` exception**: on the created-eq-1 bypass path (SKILL.md Step 3B.2), the orchestrator does NOT propagate `EDGES_*`, per-edge `EDGE_<j>_*`, `BACKLINKS_POSTED`, or `BACKLINKS_SKIPPED_NATIVE` to `output.kv`. The bypass output is one-shot-shaped тАФ `EDGES_*` keys are reserved for the multi-piece-success path. `wire-dag --no-backlinks` still emits these counters on stdout (the `BACKLINKS_*` pair is always `0` on this path because the loop is fully skipped) for stderr/log audit, but nothing reaches `output.kv`. Operators audit edge counts via `wire-dag`'s redacted stderr warnings (`EDGES_FAILED` events) and the repo-wide unavailability warning when applicable.

Changes to `emit-output`'s stdout grammar require a same-PR update to `SKILL.md` Step 4. Cycle-check semantic changes require regenerating the `test-helpers.sh` harness expectations.

### GitHub dependency-API note

The `/repos/{owner}/{repo}/issues/{number}/dependencies/blocked_by` REST endpoint surface evolved during 2024-2026 (sub-issues vs blocked-by, REST vs GraphQL surfaces). `wire-dag` is fail-open by design for the wiring action: when the probe fails (whether feature-missing or transient), all proposed edges land in `EDGES_SKIPPED_API_UNAVAILABLE`. Issue #728 added a **three-way probe classification** to disambiguate the cause: `available` тЖТ wiring proceeds; `feature-missing` (fingerprinted 404) тЖТ wiring skipped, `PROBE_FAILED=0`; `probe-failed` (5xx-after-retry, no-status-after-retry, 429, or other non-fingerprinted 4xx) тЖТ wiring skipped, `PROBE_FAILED=1` and the new transient-probe stderr fires. In the **default mode** (no `--no-backlinks`), back-links via plain `gh issue comment` still proceed even when the probe fails on either path. In `--no-backlinks` mode (created-eq-1 bypass), the back-link loop is skipped regardless of probe outcome and stderr is mode-aware (see the wire-dag stderr contract above).

**Retry policy**: one network-only retry, scoped to `5xx` and empty-status (no HTTP response at all). `429` (rate-limit), other `4xx` (e.g. `403`), and `2xx`/fingerprinted-`404` are single-attempt тАФ clear HTTP responses are not transport blips, and retrying `429` would not respect the rate-limit window cheaply. The retry was a contested decision (DECISION_1, voted 2-1 in dialectic adjudication); the alternative was pure single-attempt classification.

**Empty-probe-target path**: when `--no-backlinks` is set and `CHILDREN_FILE` is empty, `probe_target` resolves to empty string, no probe is issued, `api_available` stays `false`, and `PROBE_FAILED` stays `0`. There is no transient-probe warning on this path because no probe was attempted.

### Issue #718 transitive-closure traversal тАФ residual-risk notes

The transitive `blocked_by` traversal added for issue #718 has two documented residual risks (both surfaced and exonerated 2-1 in the design-phase voting panel):

- **Run-global truncation flag**: `_wd_traversal_truncated` is set once per run, not per-candidate. After the cap fires, every subsequent candidate whose `check-cycle` answers `CYCLE=false` is bucketed as `EDGES_FAILED` with `bound-exhausted`, even if that candidate's own reachable region was fully explored before the cap was hit. This is the fail-closed posture (DECISION_1, voted 3-0): a `CYCLE=false` answer on a known-incomplete TSV cannot be trusted, and per-candidate truncation accounting was deemed materially more complex than this fix needs (DECISION_2, voted 2-1). Operators seeing collateral `bound-exhausted` failures should raise `WIRE_DAG_TRAVERSAL_NODE_CAP` for that run.

- **Degraded individual-lookup fail-open**: when a single node's `blocked_by` lookup fails transiently (e.g., GitHub blip), the cache stores `_GH_FAIL_` and the node is treated as "no blockers" for the rest of the run. If that node is the sole intermediary that proves a cycle for some candidate, the cycle is missed silently. This preserves the file's pre-existing fail-open posture for transient API blips and avoids inflating `EDGES_FAILED` on every network hiccup; the residual cycle-miss risk is bounded to candidates whose path runs through the failed-lookup node.
